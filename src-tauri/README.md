# src-tauri 구조

Tauri 데스크탑 전용 기능을 둔다.

이 레포의 기본 화면은 Next.js 회원 웹 앱이다. Tauri는 그 화면을 데스크탑에서 실행하면서 아래 기능을 더한다.

- 버블 창과 창 제어
- 개인 관리 폴더 선택과 감지
- 로컬 SQLite 캐시
- 개인 에이전트 단기기억
- 위젯 사용 기록
- 타이머 복구
- 로컬 백업과 복구

## 데스크탑 버블 위젯 기준

Tauri 위젯은 회원 웹 앱 안의 카드나 미리보기 보드가 아니다.
Next.js 회원 웹 앱을 WebView로 띄운 뒤, 데스크탑 위에 따로 떠 있는 개인 작업 인터페이스다.

위젯 기능은 v20 디자인보드와 Tauri 위젯 4상태 명세를 기준으로 유지한다.
Tauri 위젯은 v20에 있던 기능 세트를 그대로 구현 대상으로 본다.
여기서 v20은 단순 시각 참고가 아니라 Tauri 위젯의 기능 체크리스트다.
20v에 있던 기능은 같은 책임으로 가져오고, 빠진 부분만 추가한다.
다른 회원앱 화면을 천천히 손보더라도 Tauri 위젯은 이 기능 동등성 기준을 먼저 맞춘다.
현재 화면이나 코드에서 빠져 보이는 기능은 삭제된 기능이 아니라 누락 항목으로 보고 추가한다.
앞으로 위젯을 손볼 때는 기존 기능을 줄이는 방식이 아니라, 빠진 기능을 채우는 방식으로 간다.
시각 톤과 세부 배치는 천천히 다듬더라도, 기능 범위는 v20보다 작아지면 안 된다.
우선순위는 Tauri 위젯의 v20 기능 동등성을 먼저 맞추고, 나머지 회원앱 화면과 세부 디자인은 이후 단계에서 천천히 손보는 것이다.
단, 기획에서 삭제된 받을돈/페이 위젯은 되살리지 않는다.

유지해야 할 위젯 단위는 TODO, 에이전트, 소통, 타이머, 메모, 일정, 자료, 알림이다.
상태는 기본, 반투명, 고스트, 최소화를 모두 고려한다.
창 제어는 always-on-top, 프레임리스 이동, 위치와 모드 기억, 고스트 클릭 투과, 전역 단축키, 트레이, Bubble Bar 또는 Dock Orb 최소화를 포함한다.
각 버블은 독립적인 데스크탑 위젯 창/단위다.
여러 버블을 한 창 안에 세로 카드로 쌓지 않는다.
다만 모든 버블은 하나의 Bubli Tauri 앱 생명주기에 묶인다.
버블 창은 별도 앱이나 별도 프로세스처럼 취급하지 않는다.
Bubli 하이브리드 앱이 실행될 때 필요한 버블 창을 같은 앱 실행 단위 안에서 열고 복구한다.
앱 종료 시 열려 있거나 최소화된 모든 버블 창도 함께 종료되는 구조를 기준으로 한다.

### v20 기능 동등성 기준

Tauri 위젯 작업은 "새 디자인으로 줄여서 다시 만들기"가 아니다.
v20에 있던 기능을 동등하게 가져오고, 구현이 비어 있는 부분만 채운다.
즉, 디자인 다이어트는 웹앱 화면과 표현을 정리하는 일이고, Tauri 위젯 기능을 줄이는 근거가 아니다.
회원앱 화면 정리나 디자인 다이어트가 진행되더라도 아래 항목은 위젯 범위에서 빠지지 않는다.

구현 체크리스트는 아래를 기준으로 둔다.

- 버블 단위: TODO, 에이전트, 소통, 타이머, 메모, 일정, 자료, 알림
- 표시 상태: 기본, 반투명, 고스트, 최소화
- 항목 동작: 확인, 고정, 숨김, 세부 열기
- 창 동작: 드래그 이동, 위치 기억, always-on-top, 프레임리스 창
- 고스트 동작: 낮은 시각 강조, 클릭 투과
- 최소화 동작: Bubble Bar 또는 Dock Orb
- 시스템 동작: 트레이, 전역 단축키, 앱 재시작 후 상태 복구
- 데이터 동작: 서버 원본 조회, Tauri SQLite 표시 캐시, 로컬 사용 이벤트, 집계 후 서버 반영

소통 알림 버블은 Tauri 하이브리드 앱에서 프로젝트룸 소통 전체를 작은 카드로 복사하지 않는다.
버블에는 새 댓글이나 자료 변경 같은 신호만 남기고, 실제 대화는 회원 앱의 `https` 주소로 넘긴다.
사용자가 해당 링크를 열면 그 소통 항목은 버블 안에서 숨김 상태로 접히는 흐름을 기준으로 한다.
이후 서버 API가 연결되면 같은 동작은 `PATCH /api/widget/items/{id}/state`의 숨김 상태로 이어진다.

현재 `src/lib/tauri/commands.ts`의 `TAURI_COMMANDS`와 `PLANNED_TAURI_COMMANDS`는 이 기준을 코드에 남기는 자리다.
Rust `invoke_handler`에 등록하고 프론트 adapter가 실제로 소비하는 명령은 `TAURI_COMMANDS`와 `tauriCommands`에 둔다.
아직 Rust 구현과 capability 설정이 들어가지 않은 새 명령만 planned command로 남긴다.
프론트 라우트에서 planned command를 바로 호출하지 않는다.

현재 Rust에 연결한 IPC는 위젯 창 상태 스캐폴딩과 로컬 전용 기능이다.

- 현재 위젯 창 상태 조회
- 실제 데스크탑 위젯 창 열기, 접기, 토글
- 기본, 반투명, 고스트, 최소화 모드 변경
- 위치 입력
- always-on-top 상태 입력
- 클릭 투과 상태 입력
- Dock Orb 표시 상태 입력
- 트레이 표시 상태 입력
- 전역 단축키 문자열 입력
- 위젯 창 위치와 런타임 상태 app config 저장/복원
- 로컬 SQLite 백업, 복구, 무결성 점검, 타이머 복구
- 개인 관리 폴더 선택, 스캔, 감시, 로컬 파일 검색, sync outbox 확인
- 활동 맥락 읽기
- 위젯 사용 이벤트 기록, 일별 집계, 서버 반영 대기열 적재

이 단계는 v20 기능 체크리스트를 줄이지 않고 명령 계약을 먼저 고정하는 작업이다.
실제 별도 위젯 창은 `bubli-widget` 라벨로 만들고 `/desktop-widget`을 연다.
현재 라벨은 버블 타입별로 `bubli-widget-{bubbleType}` 형태를 쓴다.
위젯 창 URL은 `/desktop-widget?bubble=todo`처럼 타입을 넘긴다.
명시적으로 닫은 버블 창은 WebView를 해제한다. 최소화와 토글 숨김은 기존처럼 창 상태를 유지한다.
앱 종료 시에는 `bubli-widget-*` 라벨의 열린 버블 창을 함께 정리한다.
아직 투명 window effect, 트레이 생성, 전역 단축키 실제 등록은 다음 구현 범위다.
위젯 위치는 `widget-window-layout.json`에 저장하고 앱 재시작 후 복원한다.
현재 Tauri QA 자동 시작 경로는 기본 실행에서 전체 8개 버블 창을 동시에 띄우지 않는다.
`/app/desktop/widgets?autoOpen=todo`처럼 한 버블만 자동으로 열 수 있다.
전체 다중 버블 QA는 `BUBLI_TAURI_WIDGET_QA_ALL=1`일 때만 켜진다. 이 값이 없으면 기본 실행은 메인 앱만 뜬다.

기존 Next 개발 서버를 그대로 쓰면서 Tauri 앱 실행만 확인할 때는 아래처럼 임시 config를 넘긴다.
이 명령은 `tauri.conf.json`을 수정하지 않고, 새 Next 서버도 띄우지 않는다.

```bash
npm exec tauri dev -- --no-watch --config '{"build":{"beforeDevCommand":"","devUrl":"http://localhost:3791"}}'
```

전체 버블 동시 QA가 필요하면 아래처럼 env를 붙인다.

```bash
BUBLI_TAURI_WIDGET_QA_ALL=1 npm exec tauri dev -- --no-watch --config '{"build":{"beforeDevCommand":"","devUrl":"http://localhost:3791"}}'
```

`/app/desktop/widgets`는 이 기준을 검토하는 회원앱 화면이다.
실제 떠 있는 버블 창은 Tauri window로 만든다.
프로젝트룸 화면, 자료보드, 채팅창을 위젯 안에 축소 복사하지 않는다.

프론트는 Tauri IPC를 직접 흩뿌리지 않고 `src/lib/tauri`의 래퍼를 통해 호출한다.
프론트 기능 코드에서는 Rust `invoke_handler`에 등록된 명령만 `src/lib/tauri/commands.ts`의 `tauriCommands`로 사용한다.
아직 구현 전인 명령은 `PLANNED_TAURI_COMMANDS`에만 두고, Rust 구현과 권한 설정이 들어간 뒤 실제 호출 함수로 연다.
각 IPC의 입력과 응답 계약은 `src/lib/tauri/commands.ts`의 `TauriCommandContract` 타입에 먼저 맞춘다.
새 IPC를 추가할 때는 v15 기획서 14.8의 Tauri 전용 IPC 목록과 맞춰야 하며, 서버 원본 데이터 변경은 HTTP API로 처리한다.
