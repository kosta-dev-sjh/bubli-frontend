# Bubli 프론트엔드 개발 가이드

기준 문서:

- `Bubli_최종기획_완성본_v15_DB회의반영_2026-06-24.md`
- `10_API-Design_v15_DB회의반영_2026-06-24.md`
- `Bubli_백엔드 개발_가이드_2026-06-19.md`

이 문서는 Bubli 프론트엔드에서 공개 사이트, 회원 웹 앱, Tauri 앱을 어떤 기준으로 나눌지 정리한다. 백엔드 API 구현은 `bubli-backend`에서 맡고, 이 레포는 화면, API client, WebSocket, Tauri IPC 연결을 맡는다.

프론트 feature는 백엔드 패키지를 그대로 복제하지 않는다. 화면 기준으로 나누되, API 호출과 데이터 원본 책임은 백엔드 개발 가이드의 `auth`, `user`, `project`, `resource`, `work`, `chat`, `memory`, `widget`, `voice`, `localsync`, `storage`, `activity`, `agent` 경계를 따른다.

## 1. 화면 기준

### 1.1 공개 사이트

공개 사이트는 비회원이 Bubli를 이해하고 데스크탑 앱을 내려받거나 로그인으로 이동하는 입구다.

| URL | 역할 |
|---|---|
| `/` | 서비스 소개와 핵심 흐름 |
| `/features` | 주요 기능 소개 |
| `/download` | 데스크탑 앱 다운로드 안내 |
| `/faq` | 자주 묻는 질문 |

공개 사이트에서는 회원 자료, 프로젝트룸, TODO, 채팅 데이터를 조회하지 않는다. 다운로드와 로그인 진입까지만 담당한다.

### 1.2 인증

| URL | 역할 |
|---|---|
| `/login` | 구글 OAuth 로그인 |

인증 화면은 route group으로는 `(auth)` 아래에 두지만, 실제 URL에는 `(auth)`가 들어가지 않는다.
v15 기준으로 `/signup` 화면과 로컬 이메일/비밀번호 회원가입, 로그인은 만들지 않는다.

### 1.3 회원 웹 앱

회원 웹 앱은 로그인 후 실제 업무를 처리하는 화면이다. v15 기준으로 회원 화면은 모두 `/app` 아래에 둔다.

| URL | 역할 |
|---|---|
| `/app` | 사용자 기준 대시보드 |
| `/app/project-rooms` | 프로젝트룸 목록 |
| `/app/resources` | 자료보드 |
| `/app/project-rooms/[roomId]/work` | 프로젝트룸 WBS/작업판 |
| `/app/chat` | 친구, 1:1 채팅, 프로젝트룸 채팅, 보이스챗 |
| `/app/settings` | 사용자 설정, 알림, 개인정보 동의, 위젯, 폴더 설정 |

`work` 화면은 WBS, TODO, 칸반, 타임라인을 조합하는 프로젝트룸 작업 화면이다. URL에 `wbs`나 `todo`를 따로 세우기보다 같은 작업을 여러 보기로 확인한다는 기획을 유지한다.
WBS는 Jira처럼 줄 단위 목록이 기본이다. 하위 작업은 색 라인으로 깊이와 묶음을 구분하고, 사용자가 줄 색을 바꿀 수 있어야 한다. 같은 WBS 데이터는 월·주·일 기간 보기로도 확인할 수 있어야 하며, 이 기한 기준은 일정/Google Calendar 연동과 같은 날짜를 사용한다. 칸반은 별도 페이지가 아니라 같은 작업판 안에서 전환해 보는 상태 보기다.

### 1.4 Tauri 앱

Tauri 앱은 별도 서비스를 새로 만드는 것이 아니라 배포된 HTTPS 회원 웹 앱의 `/app`을 WebView로 연다. 차이는 앱에서만 아래 기능을 붙인다는 점이다.

| 앱 전용 기능 | 위치 |
|---|---|
| 버블 위젯 | `features/widget`, `src-tauri/src/window` |
| 로컬 폴더 선택과 감지 | `features/managed-folder`, `src-tauri/src/folder_watch` |
| 로컬 SQLite 캐시 | `src-tauri/src/sqlite` |
| 개인 에이전트 원문 대화 | `src-tauri/src/sqlite` |
| 타이머 비정상 종료 복구 | `features/timer`, `src-tauri/src/sqlite` |
| 로컬 백업과 복구 | `src-tauri/src/backup` |

Tauri 메인 WebView에서는 소통 탭을 숨길 수 있다. 이 경우 채팅과 보이스챗은 `/app/desktop/communication` 같은 Tauri 전용 화면이나 버블 창에서 열고, 서버 API와 WebSocket, LiveKit 연결은 회원 웹 앱과 같은 것을 쓴다. 보이스챗은 LiveKit 연결을 안정적으로 쓰기 위해 배포된 HTTPS 회원 웹 앱 환경을 기준으로 실행한다.

## 3. API Client 작성 기준

공통 fetch client는 `src/lib/api`에 둔다.

- 모든 HTTP 요청은 Spring Boot API 서버를 기준으로 보낸다.
- 프론트와 Tauri는 에이전트 모듈이나 agent 컨테이너를 직접 호출하지 않는다.
- 공통 응답 포맷은 최신 API 설계와 백엔드 가이드 기준의 `success`, `data`, `error` 형태를 따른다.
- 실패 응답의 추적 값은 최상위 `traceId`가 아니라 `error.traceId`로 읽는다.
- API 함수는 feature별 `api/` 폴더에 둔다.
- 서버 상태값은 문자열 리터럴 타입이나 enum으로 먼저 고정한다.
- LiveKit key와 secret은 클라이언트 환경 변수에 두지 않는다.
- 백엔드 Entity를 프론트 타입으로 가정하지 않는다. 프론트 타입은 백엔드 Response DTO 기준으로 만든다.
- 일반 목록은 `PageResponse<T>`를 사용하고, 채팅과 프로젝트룸 이벤트는 `sequence` 기반 응답을 사용한다.
- WebSocket payload는 HTTP `ApiResponse<T>`가 아니라 `RealtimeEnvelope<T>` 기준으로 받는다.
- `timer start`, `timer stop`, `chat message send`, `agent job create`, `widget rollup submit`, `file sync approve` 요청에는 `idempotencyKey`를 붙인다.
- 웹 refresh token은 httpOnly cookie로만 다루고, Tauri refresh token은 OS secure storage에 둔다.

권장 분리:

```text
src/lib/api
├── client.ts
├── response.ts
└── errors.ts

src/features/project-room/api
└── projectRoomApi.ts

src/features/communication/api
├── chatApi.ts
└── voiceApi.ts
```

공통 응답 타입 예시:

```ts
export type ApiSuccess<T> = {
  success: true;
  data: T;
  error: null;
};

export type ApiFailure = {
  success: false;
  data: null;
  error: {
    code: string;
    message: string;
    traceId: string;
    fields?: { field: string; reason: string }[];
  };
};

export type ApiResponse<T> = ApiSuccess<T> | ApiFailure;
```

## 3.1 백엔드 도메인과 프론트 연결 기준

백엔드 가이드는 도메인 간 Repository/Entity 직접 접근을 금지하고 공개 Service를 통해 연결한다. 프론트도 같은 원칙을 화면 구조에 적용한다.

| 백엔드 도메인 | 프론트에서 보는 기준 |
|---|---|
| `project` | 프로젝트룸, 멤버, 친구 기반 초대의 원본 API |
| `resource` | 개인 자료와 프로젝트룸 자료, 댓글, 버전, 분석 상태의 원본 API |
| `work.task` | 개인 TODO와 프로젝트룸 TODO의 원본 API |
| `work.wbs` | WBS/작업판 원본 API |
| `work.schedule`, `personal.calendar` | 내부 일정과 외부 캘린더 표시 기준 |
| `chat` | 1:1 채팅, 프로젝트룸 채팅 메시지 원본 |
| `voice` | LiveKit room과 참가 토큰 발급 |
| `agent` | 후보 생성, agent_jobs, agent_suggestions |
| `memory` | 프로젝트룸 장기기억, 사용자가 확인한 하루정리 요약 |
| `widget` | 버블 항목 상태와 날짜별 사용 집계 |
| `localsync` | Tauri 개인 관리 폴더 이벤트의 서버 반영 |
| `activity` | 동의 기반 앱/창 제목과 머문 시간 기록 |

대시보드와 위젯은 원본 데이터를 직접 소유하지 않는다. 필요한 데이터는 각 도메인 API 결과를 조합해서 표시한다.

## 4. WebSocket과 소통 기준

채팅과 알림은 서버 DB를 원본으로 둔다. Tauri SQLite는 프로젝트룸 채팅을 빠르게 보여주기 위한 캐시다.

| Topic | 용도 |
|---|---|
| `/topic/chat/{chatRoomId}` | 채팅방 새 메시지 |
| `/topic/project-rooms/{roomId}/events` | 에이전트 작업 완료, 프로젝트룸 이벤트 |
| `/user/queue/notifications` | 개인 알림 |

웹 `/app/chat`에서는 친구, 1:1 채팅, 프로젝트룸 채팅, 보이스챗을 모두 제공한다. Tauri 전용 소통 창은 같은 API와 WebSocket 연결을 재사용한다.

채팅 메시지는 `clientMessageId`로 중복 전송을 막고, 서버가 저장한 `roomSequence`를 기준으로 표시 순서를 확정한다. 누락 메시지는 `afterSequence`, 이전 메시지는 `beforeSequence` 기준으로 조회한다.

프로젝트룸 이벤트는 `/topic/project-rooms/{roomId}/events`에서 `AGENT_JOB_STATUS_CHANGED`, `AGENT_SUGGESTIONS_CREATED`, `RESOURCE_ANALYSIS_COMPLETED` 같은 상태 변화를 받는다. 채팅 메시지 본문은 프로젝트룸 이벤트가 아니라 `/topic/chat/{chatRoomId}`에서 따로 받는다.

## 5. LiveKit 기준

보이스챗은 프로젝트룸 소통의 보조 기능이다.

- Tauri에서 채팅과 보이스챗을 열 때도 배포된 HTTPS 회원 웹 앱의 연결 기준을 사용한다.
- 클라이언트는 `/api/voice/rooms/{id}/token`으로 토큰을 받는다.
- LiveKit `serverUrl`과 접속 token만 클라이언트가 사용한다.
- API key와 secret은 서버에만 둔다.
- 보이스 참여는 프로젝트룸 멤버 권한을 기준으로 확인한다.
- token 응답은 `serverUrl`, `token`, `voiceRoomId`, `participantId`, `expiresAt` 기준으로 받는다.

## 6. Tauri IPC와 SQLite 기준

로컬 파일 접근과 SQLite는 Tauri 앱 안에서만 다룬다. 서버에 바로 보여줘야 하는 데이터는 HTTP API를 원본으로 둔다.

| 구분 | 원본 또는 저장 위치 |
|---|---|
| TODO, 일정, 알림, 채팅, 타이머 원본 | 서버 DB |
| 프로젝트룸 채팅 캐시 | Tauri SQLite `local_room_message_cache` |
| 개인 에이전트 원문 | Tauri SQLite `local_agent_messages` |
| 위젯 표시 캐시 | Tauri SQLite `local_widget_display_cache` |
| 위젯 상세 사용 이벤트 | Tauri SQLite `local_widget_usage_events` |
| 위젯 날짜별 집계 | Tauri SQLite 집계 후 서버 `widget_daily_summaries` |
| 실행 중 타이머 복구 상태 | Tauri SQLite `local_timer_state` |
| 서버 반영 대기열 | Tauri SQLite `local_sync_outbox` |
| 로컬 백업 목록 | Tauri SQLite `local_backup_manifest` |

Tauri IPC는 `src/lib/tauri`에서 래핑한다. Rust `invoke_handler`에 등록된 명령만 `tauriCommands`에서 실제 호출 함수로 열고, 아직 구현 전인 명령은 `PLANNED_TAURI_COMMANDS`에만 둔다. 이렇게 해야 웹 코드가 구현되지 않은 로컬 기능을 실수로 호출하지 않는다.

주요 IPC:

```text
app_ready
select_managed_folder
scan_managed_folder
watch_managed_folder
search_local_files
sync_room_messages
backup_local_sqlite
restore_local_sqlite_backup
check_local_sqlite_integrity
read_activity_context
record_widget_usage_event
rollup_widget_usage
sync_widget_usage_summary
flush_sync_outbox
recover_timer_state
```

## 7. 디자인 v20 구현 기술 스택

디자인 v20은 자료보드, WBS/작업판, 채팅, 버블 위젯처럼 한 화면 안에서 데이터와 상태가 자주 바뀐다. 따라서 기술 선택 기준은 화면을 빠르게 그리는 것, 긴 목록을 가볍게 다루는 것, 모션을 제한적으로 쓰는 것, Tauri 전용 기능을 웹 코드와 섞지 않는 것이다.

리서치 기준일은 2026-06-22다. 패키지는 최신 버전과 사용량, 공식 문서의 유지 상태, Next.js App Router와 React 19 조합을 기준으로 골랐다.

### 7.1 기본 UI와 디자인 시스템

| 용도 | 선택 | 쓰는 이유 | 적용 위치 |
|---|---|---|---|
| 스타일 엔진 | `tailwindcss`, `@tailwindcss/postcss` | Tailwind CSS v4는 설정이 단순하고, 디자인 토큰을 CSS 변수로 관리하기 좋다. v20의 글래스 패널, 칩, 버블 상태를 빠르게 맞출 수 있다. | `src/styles/globals.css`, `postcss.config.mjs` |
| 컴포넌트 기준 | `shadcn/ui` 방식 | 라이브러리 런타임에 종속되지 않고, 필요한 컴포넌트 코드를 프로젝트 안에 가져와 수정할 수 있다. Bubli처럼 유리 질감과 버블 톤을 직접 조정해야 하는 화면에 맞다. | `components.json`, `components/ui` |
| 컴포넌트 작업장 | `storybook`, `@storybook/nextjs-vite` | 전체 앱을 매번 열지 않고 버튼, 카드, 버블, 채팅 메시지, 후보 상태를 단위별로 확인한다. 바이브 코딩으로 빠르게 만든 컴포넌트도 기능 상태와 비기능 상태를 따로 검수할 수 있다. | `.storybook`, `*.stories.tsx` |
| 클래스 조합 | `class-variance-authority`, `clsx`, `tailwind-merge` | 버튼, 칩, 버블, 상태 배지처럼 variant가 많은 컴포넌트를 안전하게 조합한다. Tailwind 클래스 충돌도 줄인다. | `src/lib/utils.ts`, `components/ui` |
| 아이콘 | `lucide-react` | 아이콘을 SVG 컴포넌트로 가져오며 tree-shaking이 된다. Bubli 규칙상 이모지 대신 선형 아이콘을 쓰기 좋다. | `components/icons`, 버튼, 탭, 상태 표시 |
| 알림 토스트 | `sonner` | 작업 저장, 후보 승인, 채팅 연결 상태 같은 짧은 피드백을 가볍게 보여준다. | `src/app/providers.tsx` |

운영 기준:

- 공통 컴포넌트는 shadcn/ui를 그대로 쓰지 않고 Bubli 토큰으로 감싼다.
- 버블과 패널은 CSS 변수 기반으로 만든다. 화면마다 임의 색을 만들지 않는다.
- 이모지는 쓰지 않는다. 기능 표시는 `lucide-react` 아이콘과 텍스트를 같이 쓴다.
- Storybook story에는 빈 상태, 긴 텍스트, 확인 필요 항목, 후보, 승인 전, 승인됨, 고스트 버블 같은 기획 상태를 같이 둔다.
- Storybook은 실제 API 원본을 대신하지 않는다. 서버 DB 원본, Tauri SQLite 캐시, 개인 에이전트 로컬 원문 같은 책임 경계는 story 설명과 타입 이름으로 드러낸다.

### 7.2 서버 상태와 클라이언트 상태

| 용도 | 선택 | 쓰는 이유 | 적용 위치 |
|---|---|---|---|
| 서버 상태 | `@tanstack/react-query` | 프로젝트룸, 자료, TODO, 알림, 채팅 목록처럼 서버 원본 데이터를 캐시하고 재조회한다. 화면 이동 때 같은 데이터를 반복 요청하지 않게 한다. | `src/lib/query`, `features/*/api` |
| 클라이언트 상태 | `zustand` | 사이드바 열림, 선택한 프로젝트룸, 버블 표시 상태처럼 서버 원본이 아닌 UI 상태를 작게 관리한다. Redux처럼 구조가 커지지 않는다. | `src/stores` |
| 폼 | `react-hook-form`, `zod`, `@hookform/resolvers` | 프로젝트룸 생성, 자료 업로드, 위젯 설정처럼 검증이 있는 폼을 타입과 함께 관리한다. 로그인은 Google OAuth 진입으로 처리하고 자체 이메일/비밀번호 입력 폼을 만들지 않는다. | `features/*/components`, `lib/validators` |

운영 기준:

- 서버 DB가 원본인 값은 React Query로 읽고 갱신한다.
- 위젯 위치, 패널 열림, 현재 탭처럼 화면만을 위한 값은 Zustand에 둔다.
- React Query 캐시를 직접 영구 저장소처럼 쓰지 않는다.
- API 응답 검증이 필요한 곳은 `zod` schema를 두고, 백엔드 DTO 확정 후 타입을 맞춘다.

### 7.3 자료보드, WBS, 채팅 성능

| 용도 | 선택 | 쓰는 이유 | 적용 위치 |
|---|---|---|---|
| 테이블/목록 로직 | `@tanstack/react-table` | 자료보드 필터, 정렬, 선택, 페이지네이션을 UI와 분리해서 다룬다. 디자인 v20처럼 직접 만든 카드형 목록에도 맞다. | `features/resources`, `features/wbs` |
| 긴 목록 가상화 | `@tanstack/react-virtual` | 자료, 채팅, 알림, WBS 항목이 많아져도 보이는 row만 렌더링한다. 렉 방지 핵심이다. | 자료보드 목록, 채팅 메시지, WBS 트리 |
| 드래그 앤 드롭 | `@dnd-kit/core`, `@dnd-kit/sortable`, `@dnd-kit/utilities` | WBS 순서 변경, 칸반 이동, 대시보드 카드 배치, 버블 위치 조정에 쓴다. 접근성과 센서 제어가 좋다. | `features/wbs`, `features/dashboard`, `features/widget` |
| 모션 | `motion` | 버블 등장, 접기, 패널 전환 같은 짧은 모션에만 쓴다. 대량 목록이나 채팅 메시지 전체에는 쓰지 않는다. | `components/bubbles`, 주요 전환 |
| 날짜 처리 | `date-fns` | 일정, D-day, 타이머, 하루정리 표시를 가볍게 처리한다. | `features/calendar`, `features/timer` |

렉 방지 기준:

- 자료보드와 채팅은 처음부터 가상화를 전제로 컴포넌트를 만든다.
- WBS/칸반에서 드래그 중에는 무거운 API 호출을 하지 않는다. 화면 상태를 먼저 바꾸고, 저장은 API mutation으로 분리한다.
- 애니메이션은 `opacity`, `transform` 중심으로 쓴다. `width`, `height`, `top`, `left`를 계속 바꾸는 모션은 피한다.
- `motion`은 화면 전체 장식보다 상태 변화 피드백에 쓴다.
- 모바일과 Tauri WebView에서는 `prefers-reduced-motion`을 존중한다.

### 7.4 채팅, 보이스챗, 실시간 연결

| 용도 | 선택 | 쓰는 이유 | 적용 위치 |
|---|---|---|---|
| STOMP/WebSocket | `@stomp/stompjs` | Spring WebSocket topic 구조와 맞다. `/topic/chat/{chatRoomId}`, `/user/queue/notifications` 연결을 명확히 다룬다. | `lib/websocket`, `features/communication` |
| 보이스챗 | `@livekit/components-react`, `livekit-client` | LiveKit 공식 React 컴포넌트와 client SDK를 사용한다. 웹 `/app/chat`과 Tauri 전용 소통 창이 같은 토큰 발급 API를 쓴다. | `features/communication` |

운영 기준:

- LiveKit token은 클라이언트에서 만들지 않는다. API 서버에서 받은 `serverUrl`, `token`만 사용한다.
- Tauri에서도 채팅/보이스는 배포된 HTTPS 회원 웹 앱 연결 기준을 따른다.
- 메시지 전송 완료 기준은 로컬 optimistic UI가 아니라 서버 저장 후 WebSocket으로 돌아온 메시지다.
- 채팅 메시지가 많아지는 화면은 `@tanstack/react-virtual`을 같이 쓴다.

### 7.5 Tauri 전용 기능

| 용도 | 선택 | 쓰는 이유 | 적용 위치 |
|---|---|---|---|
| 파일/폴더 선택 | `@tauri-apps/plugin-dialog` | 개인 관리 폴더 선택과 파일 선택은 브라우저 권한으로 해결하기 어렵다. | `src/lib/tauri`, `src-tauri/src/commands` |
| 로컬 파일 접근 | `@tauri-apps/plugin-fs` | 개인 관리 폴더 scan/watch 결과를 앱 안에서 다룰 때 필요하다. | `features/managed-folder`, Tauri commands |
| 로컬 SQLite | `@tauri-apps/plugin-sql` | 프로젝트룸 채팅 캐시, 개인 에이전트 원문, 위젯 사용 이벤트, local_sync_outbox를 로컬에 둔다. | `src-tauri/src/sqlite` |
| 시스템 알림 | `@tauri-apps/plugin-notification` | 데스크탑 알림, 타이머, 소통 알림에 쓴다. | `features/notification`, Tauri commands |

운영 기준:

- Tauri plugin은 JavaScript 패키지만으로 끝나지 않는다. 실제 기능 구현 시 Rust 쪽 plugin 등록과 capability 설정을 함께 해야 한다.

- 웹에서 필요한 기능은 HTTP API로 먼저 설계하고, Tauri 전용 기능만 IPC로 뺀다.

## 8. 구현 우선순위

1. v15 기준 라우트와 레이아웃을 먼저 만든다.
2. 디자인 v20에서 옮길 수 있는 공통 UI, 도메인 카드, 버블 컴포넌트를 Storybook에 분리한다.
3. `src/lib/api`, `src/types/api`, `features/*/api`의 공통 응답/에러 타입과 v15 API client를 만든다.
4. 공개 사이트, 로그인, `/app` 대시보드의 빈 화면을 만든다.
5. 프로젝트룸, 자료보드, WBS/작업판은 Storybook에서 검수한 컴포넌트로 조립한다.
6. 채팅/보이스챗은 `/app/chat`에서 먼저 붙이고, Tauri 전용 창은 같은 연결을 재사용한다.
7. Tauri IPC는 화면이 붙은 뒤 로컬 폴더, SQLite, 타이머, 위젯 순서로 연결한다.
8. 버블 위젯은 회원 웹 화면을 작게 복제하지 않고, 개인 작업 인터페이스로 따로 구성한다.

## 9. 확인 기준

- `/app` 밖에 회원 전용 라우트가 남아 있지 않아야 한다.
- `/app/project-rooms/[roomId]/work` 안에서 WBS, TODO, 칸반, 타임라인을 조합해야 한다.
- 프론트와 Tauri가 agent를 직접 호출하는 코드가 없어야 한다.
- Tauri SQLite에만 있어야 하는 개인 에이전트 원문과 위젯 상세 이벤트가 서버 API로 전송되지 않아야 한다.
- 채팅과 보이스챗은 웹과 Tauri 전용 창에서 같은 API 경계를 사용해야 한다.
- `npm run storybook`에서 공통 UI, 자료 카드, 후보 카드, 작업 카드, 채팅 메시지, 버블 상태를 단위별로 확인할 수 있어야 한다.

### 9.1 Storybook 점검 순서

Storybook은 디자인 완성본을 보는 곳이 아니라, 기획 상태와 화면 조각이 빠지지 않았는지 먼저 확인하는 작업장이다. 1차 점검은 기능 누락과 책임 경계를 보고, 디자인보드 v20 정합화는 그 다음 단계에서 한 번에 맞춘다.

점검 순서는 아래처럼 잡는다.

| 순서 | Storybook에서 볼 묶음 | 확인할 것 |
|---|---|---|
| 1 | `UI`, `Domain`, `Bubbles` | 버튼, 상태 배지, 자료 카드, 작업 카드, 후보 카드, 버블 카드가 Bubli 토큰과 기본 밀도를 쓰는지 본다. |
| 2 | `Features/Resources`, `Features/ProjectRoom`, `Features/WBS`, `Features/Todo` | 자료 업로드, 후보 확인, WBS/TODO 연결, 프로젝트룸 초대와 권한 흐름이 v15 기획과 맞는지 본다. |
| 3 | `Features/Agent`, `Features/Communication` | 에이전트는 후보 생성과 job 상태만 보여주고, 확정 데이터 직접 변경처럼 보이지 않는지 본다. 채팅/보이스는 API 서버와 LiveKit 토큰 경계를 유지해야 한다. |
| 4 | `Features/Widget`, `Features/Timer`, `Features/Settings`, `Features/ManagedFolder` | 서버 원본, Tauri SQLite 캐시, 로컬 대기열, 타이머 복구, 위젯 상세 이벤트의 저장 위치가 섞이지 않는지 본다. |
| 5 | `Features/PublicSite`, `Features/Auth`, `Features/Download` | 공개 사이트, Google OAuth 진입, 다운로드 안내가 회원 앱 기능과 섞이지 않는지 본다. |

Storybook 점검에서 바로 고쳐야 하는 것은 기획 결정과 충돌하는 부분이다.

- 비회원 임시 참여 기능이 보이면 제거한다.
- 자체 계정 입력으로 가입/로그인하는 흐름처럼 보이면 Google OAuth 기준으로 바꾼다.
- 프로젝트와 프로젝트룸을 별도 서비스처럼 나누면 프로젝트룸 기준으로 합친다.
- 프론트나 Tauri가 에이전트 서버를 직접 호출하는 것처럼 보이면 API 서버 경유로 정리한다.
- 버블이 프로젝트룸 화면의 축소판처럼 보이면 개인 작업 인터페이스로 다시 잡는다.

반대로 색, 여백, 글래스 질감, 모션 세부값은 1차 컴포넌트 점검 뒤 `design-board-v20-polish` 단계에서 한 번에 맞춘다.

## 10. 백엔드 연동 확인 기준

프론트와 백엔드는 API 기준으로 작업을 맞춘다. 화면 명칭이나 feature 폴더명이 달라도 실제 통신은 최신 `10_API-Design.md`의 `/api/...` 경로와 백엔드 개발 가이드의 도메인 책임을 따른다.

### 10.1 API 계약 원칙

- API 경로는 최신 `10_API-Design.md`를 우선한다.
- v15 기획서, 백엔드 개발 가이드, API 계약 확정안이 더 구체적인 구현 기준을 제시하면 더 구체적인 계약을 프론트 API client에 반영한다.
- 인증 endpoint와 토큰 저장 방식은 2026-06-22 API 계약 확정안을 기준으로 한다.
- 프론트 API 함수는 백엔드 Controller 이름이 아니라 화면 feature 기준으로 둔다.
- 요청/응답 타입은 백엔드 Entity가 아니라 Response DTO 기준으로 만든다.
- 실패 응답은 `error.code`, `error.message`, `error.traceId`를 기준으로 처리한다.
- 채팅, 보이스챗, 알림, 에이전트 job처럼 실시간성이 있는 기능은 HTTP API와 WebSocket/LiveKit 연결을 함께 확인한다.

### 10.2 API 수동 검증 파일 매칭

백엔드 가이드는 IntelliJ HTTP Client의 `.http` 파일을 권장한다. 프론트는 같은 묶음 기준으로 API client 파일을 만든다.

| 백엔드 검증 파일 | 프론트 API 위치 | 확인할 기능 |
|---|---|---|
| `auth.http` | `features/auth/api` | 구글 OAuth 로그인, refresh, 로그아웃, 내 정보 조회 |
| `project.http` | `features/project-room/api` | 프로젝트룸, 멤버, 친구 기반 초대 |
| `resource.http` | `features/resources/api` | 개인 자료, 프로젝트룸 자료, 다운로드 URL, 버전, 댓글 |
| `personal.http` | `features/dashboard/api`, `features/timer/api`, `features/notification/api`, `features/calendar/api` | 대시보드, 타이머, 알림, 일정 |
| `work.http` | `features/todo/api`, `features/wbs/api` | TODO, WBS, 작업판, 후보 승인 |
| `agent.http` | `features/agent/api` | agent_jobs, 후보 생성, 제안함 |
| `chat.http` | `features/communication/api` | 친구, 채팅방, 메시지, agent-command |
| `memory.http` | `features/communication/api`, `features/agent/api` | 프로젝트룸 장기기억, 하루정리 요약 |
| `widget.http` | `features/widget/api` | 위젯 표시 데이터, 항목 상태, 사용 집계 |
| `voice.http` | `features/communication/api` | LiveKit room, token, leave, end |
| `localsync.http` | `features/managed-folder/api`, `features/activity/api` | 개인 관리 폴더, 로컬 파일 이벤트, 활동 기록 |

### 10.3 소통 기능 API 계약

소통 화면은 여러 백엔드 도메인을 함께 쓴다. 프론트에서는 `features/communication` 안에서 조합하되, 원본 책임은 아래처럼 나눈다.

| 기능 | HTTP API | 실시간 연결 | 백엔드 원본 |
|---|---|---|---|
| 친구 목록 | `GET /api/friends` | 없음 | `project`/친구 관계 |
| Bubli ID 검색 | `GET /api/friends/search?bubliId={id}` | 없음 | `project`/친구 관계 |
| 친구 요청 | `GET /api/friend-requests`, `POST /api/friend-requests`, `PATCH /api/friend-requests/{id}/accept`, `PATCH /api/friend-requests/{id}/reject` | 알림은 `/user/queue/notifications` | `project`/친구 관계 |
| 1:1 채팅방 | `POST /api/chat/direct-rooms`, `GET /api/chat/rooms` | `/topic/chat/{chatRoomId}` | `chat` |
| 채팅 메시지 | `GET /api/chat/rooms/{id}/messages`, `POST /api/chat/rooms/{id}/messages`, `PATCH /api/chat/rooms/{id}/read` | `/topic/chat/{chatRoomId}` | `chat_messages` |
| 프로젝트룸 에이전트 명령 | `POST /api/project-rooms/{roomId}/agent-command` | `/topic/project-rooms/{roomId}/events`, `/topic/chat/{chatRoomId}` | `chat`이 받고 `agent` 공개 Service 호출 |
| 프로젝트룸 장기기억 | `POST /api/project-rooms/{roomId}/memory-summaries`, `GET /api/project-rooms/{roomId}/memory-summaries` | `/topic/project-rooms/{roomId}/events` | `memory` |
| 보이스챗 | `POST /api/voice/rooms`, `GET /api/voice/rooms/{id}`, `POST /api/voice/rooms/{id}/token`, `PATCH /api/voice/rooms/{id}/leave`, `PATCH /api/voice/rooms/{id}/end` | LiveKit Client SDK | `voice`, LiveKit |

소통 기능에서 프론트가 직접 저장 원본을 판단하지 않는다. 메시지는 서버 DB 저장 성공 후 WebSocket으로 받는 값을 기준으로 전송 완료 처리한다. Tauri 채팅 캐시는 서버 메시지를 빠르게 보여주기 위한 복제본이다.
