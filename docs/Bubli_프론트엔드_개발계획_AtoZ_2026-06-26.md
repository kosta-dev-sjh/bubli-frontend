# Bubli 프론트엔드 개발계획 (A–Z)

작성일: 2026-06-26 · 담당: 프론트엔드/타우리 · 성격: 착수~완성 전체 계획
이 문서가 프론트 작업의 단일 기준이다. 과거 "디자인 전면개편 계획"은 이 문서로 대체한다. 짝 문서: `Bubli_타우리_구현스펙_로컬자료검색`(Tauri 코드 스펙), `Bubli_프론트타우리_벤치마킹_구현계획`(벤치마킹 배경). 단계별 실행 순서는 같은 폴더의 `Bubli_프론트엔드_바이브코딩_가이드.csv`를 따른다.

## 0. 지금 상태 (냉정하게)

진척이 안 나 보이는 건 정상이다. 최근 작업이 "부품 생산"이 아니라 "이미 붙은 점검/정책 패널 청소"에 머물렀기 때문이다. 현재 체감 진척:

- 1차 컴포넌트 추출: 약 80% (스토리 144개, 다수가 경계·정책 검증 패널)
- 라우트 연결: 약 90%
- Storybook 대표 점검: 약 74%
- 디자인보드 v20 정합화: 약 49%
- API 계약 반영: 약 20%
- Tauri 경계 검증: 약 43%

문제는 80%에서 "더 뽑기"가 아니라 "정리"로 새고 있었다는 점이다. 그래서 이 계획은 **순서를 강제로 다시 박는다.** 청소는 잠시 멈춘다.

진행 원칙 한 줄: **부품을 먼저 다 깐다 → 전 화면이 일단 다 보이게 한다 → 그다음 디자인을 v20으로 크게 갈아엎는다 → API는 확정되면 붙인다.** 이 순서를 어기지 않는다.

## 2. P0 — "1차 컴포넌트 추출 완료"의 정의와 작업

지금 "80%"가 애매한 이유는 완료 기준이 없어서다. 기준을 박는다.

### 2.1 완료 기준 (이 5개를 다 만족하면 "추출 완료")

1. IA(`03_IA-Screen-Structure`)의 모든 잎 화면이 실제 라우트에 마운트돼 있다.
2. 각 화면이 최소 1개의 컴포넌트로 골격이 잡혀 있다.
3. 각 컴포넌트가 Storybook 스토리를 가진다.
4. 각 화면이 빈/로딩/에러/정상 4상태를 가진다(목 데이터 허용).
5. 중복·정책·검증 패널은 삭제하지 않고 `점검용`으로 태그만 단다.

### 2.2 분류 작업 (삭제 금지, 라벨만)

스토리 144개를 세 통으로 나눈다. `docs/component-inventory.csv`로 산출한다.

- `완료`: IA 잎 화면을 직접 그리는 대표 컴포넌트.
- `미완료`: IA에 있는데 화면/컴포넌트가 없는 것.
- `점검용`: 같은 화면의 보조 정책·경계·복구·API 설명 패널. 화면에서 떼더라도 Storybook에는 남긴다.

분류 기준 컬럼: 스토리ID / feature / IA화면 / 분류 / 라우트마운트(Y/N) / 4상태(Y/N) / 비고.

### 2.3 누락 화면 (IA 대조 결과 — 이미 확정)

라우트 대부분은 있다. 아래만 채우면 IA가 닫힌다.

| 누락/보강 | 결정 | 위치 |
|---|---|---|
| 일정(캘린더) 화면 | 신규 라우트 추가 | `/app/calendar` + 대시보드 위젯 |
| 에이전트 제안함(승인 인박스) | 신규 화면 | `/app/project-rooms/[roomId]/work` 탭 + 전역 제안함 |
| 대시보드 구성요소 | 조립 확인 | 오늘 TODO·마감·확인필요·타이머·룸별요약 카드 |
| WBS 4보기(트리/칸반/타임라인/간트) | 보기 토글로 묶음 | `/work` 내부 |
| 데스크탑 위젯 8종 버블 | 골격 확인 | `/app/desktop/widgets` |

이 표가 P0에서 "추가로 뽑을 부품" 목록이다. 그 외에는 새로 뽑지 않는다.

### 2.4 P0 산출물

`docs/component-inventory.csv`(144개 분류), 누락 5종 골격 추가, 그리고 README에 "1차 컴포넌트 추출 완료" 선언 한 줄.

## 3. P1 — 전 화면 노출

모든 라우트를 끝까지 클릭되게 만든다. 디자인은 아직 신경 쓰지 않는다.

- 회원 앱 좌측 내비에서 모든 IA 화면으로 이동 가능.
- 각 화면은 목 데이터로 정상 화면 + 빈/로딩/에러 상태를 가진다.
- 화면 본체에는 `점검용` 패널을 붙이지 않는다(스토리북에만 둔다).
- 라우트 가드: `/app` 밖에 회원 업무 화면을 만들지 않는다.

끝 판정: 로그인→대시보드→프로젝트룸→자료보드→작업판→소통→일정→설정→위젯까지 한 번에 클릭되며 어디서도 안 깨진다.

## 4. P2 — 디자인 전면 개편 (v20 기준)

핵심: 화면을 새로 그리지 않는다. 컴포넌트의 겉만 v20 톤으로 바꾼다. 구조(props·데이터·라우트)는 P0/P1 자산을 그대로 쓴다.

### 4.1 디자인 시스템 4층 (아래에서 위로)

1. 토큰: `styles/globals.css`(8034줄)를 토큰 파일로 정리하고 Tailwind v4 `@theme`로 노출. 색은 CLAUDE.md 8장 값만.
2. 기본 컴포넌트(`components/ui`): Button, Input, Card, Panel, Pill, Tab, Toggle, Modal, Bubble. 유리/버블 질감을 여기서 한 번만 구현.
3. 도메인 카드(`components/domain`, `components/bubbles`): 자료 카드, 후보 카드, 작업/WBS 카드, 채팅 버블, 위젯 버블.
4. 화면: P1 화면을 위 컴포넌트로 재조립.

### 4.2 진행 방식

Storybook 우선. 컴포넌트가 스토리북에서 통과하면 화면에 끼운다. v20 디자인보드 계보(유리·파스텔, Rain Glass)를 기준으로 한다. 하드코딩 색 0개가 통과 기준이다.

### 4.3 톤

v20 유리·파스텔을 유지·발전한다(투명감·광택 고도화). 현재 디자인 리서치를 거쳐 **새 디자인 버전(과거 v21 폐기 후 신규)**이 별도 트랙에서 제작 중이다. P2는 그 새 버전이 확정되는 대로 토큰·primitive·질감의 기준으로 삼는다. 즉 디자인 톤의 단일 소스는 "제작 중인 새 디자인 버전"이고, 이 개발계획은 그것을 코드로 옮기는 일을 맡는다. 다만 디자인 확정이 늦어도 P2의 토큰 분해·구조 작업(globals.css 정리, primitive 골격)은 톤과 무관하게 먼저 진행한다 — 색·질감만 나중에 끼우면 되도록.

## 5. P3 — API 연동

### 5.1 원칙

디자인이 흔들리면 로딩·에러 UI도 바뀐다. 그래서 P2(토큰+기본 컴포넌트)가 선 다음 화면 단위로 붙인다. 한꺼번에 14개 feature를 다 붙이지 않는다.

### 5.2 계약 원본

백엔드는 Swagger가 없다. 계약은 `bubli-backend/docs/http/*.http` 9개 + `ApiResponse.java`(공통 봉투 `success/data/error`)다. 프론트 `apiRequest`는 이미 이 봉투를 따른다. 백엔드 develop 최신화(`git pull origin develop`) 먼저.

### 5.3 연동 순서 (의존성 기준)

1. 인증(모든 화면 전제) → 2. 프로젝트룸·자료(`/api/resources`) → 3. 작업(WBS·TODO) → 4. 소통(채팅 STOMP·보이스 LiveKit) → 5. 에이전트(`agent_jobs`·`agent_suggestions`) → 6. 위젯·타이머·활동·일정.

각 화면은 (a)디자인 안정 + (b)해당 엔드포인트 develop 머지, 둘 다 충족될 때 붙인다. 안 되면 목으로 두고 넘어간다.

## 6. PX — Tauri 네이티브 (P1부터 병행) + 벤치마킹 기술 적용

상세 코드 스펙은 `Bubli_타우리_구현스펙_로컬자료검색`에 있다. 여기선 벤치마킹 기술이 어디 들어가는지만.

| 벤치마킹 출처 | 우리 적용 | 들어가는 화면/모듈 |
|---|---|---|
| Docufinder: 로컬 FTS5 본문 검색 | 개인 폴더를 업로드 없이 내용 검색 | `src-tauri/search`+`index`, `features/resources` 검색창 |
| Docufinder: 폴더 실시간 감시 | 추가/수정/삭제 자동 재색인 | `src-tauri/folder_watch`, `features/managed-folder` |
| Docufinder: 텍스트 추출(Rust) | 가벼운 형식 로컬 추출(txt/md/docx/pdf) | `src-tauri/extract` |
| Docufinder: "AI 끄면 오프라인" | 개인 원문 비업로드 프라이버시 메시지 | 랜딩·설정 문구 |
| Drifty: 활동 인식 | 활성 앱·창·시간 → 작업 맥락 제안 | `src-tauri/activity`, `features/activity` |

핵심 결정(확정): 무거운 AI(임베딩·RAG·OCR)는 서버(pgvector), 로컬 즉시 검색·폴더 감시·가벼운 추출·활동만 Tauri. 자료보드 검색창에서 둘을 점진 병합(로컬 먼저, 서버 의미 뒤).

## 8. 바이브코딩 방법론 (이 프로젝트에 맞춘 실전)

쌩초보 기준으로, "AI에게 시켜서 만든다"를 안전하게 굴리는 법.

### 8.1 한 작업의 표준 루프

1. 작은 단위로 쪼갠다(한 컴포넌트/한 화면/한 IPC).
2. 컨텍스트를 준다: 관련 기준 문서 경로 + 디자인 토큰 + 예시 컴포넌트 1개.
3. 만들게 한다 → 스토리북/브라우저로 눈으로 본다.
4. 검증 4종(`lint`/`typecheck`/`build`/`cargo check`)을 통과시킨다.
5. 작은 커밋. 다음 작업.

### 8.2 좋은 프롬프트 골격 (복붙용)

```
역할: 너는 Bubli 프론트 개발자다. 기준 문서를 먼저 읽어라:
- /Users/maren/EDU/Final Project/CLAUDE.md (규칙·디자인 토큰)
- bubli-frontend/docs/PROJECT_STRUCTURE.md (폴더·라우트)
- (관련) wiki 03_IA-Screen-Structure / 10_API-Design

작업: <한 가지만> 예) features/calendar에 월간 일정 화면 골격을 만든다.
제약:
- 색은 CLAUDE.md 8장 토큰만. 이모지 금지. 아이콘은 인라인 SVG(stroke,currentColor).
- 빈/로딩/에러/정상 4상태를 만든다. API는 아직 붙이지 말고 목 데이터.
- 기존 components/ui를 재사용한다. 새 색·새 라이브러리 추가 금지.
산출: 컴포넌트 + 스토리(.stories.tsx). 끝나면 npm run typecheck 통과 확인.
```

프롬프트 원칙: (1) 기준 문서 경로를 매번 준다, (2) 한 번에 하나만, (3) 제약(토큰·금지)을 명시, (4) 산출물과 검증을 못박는다, (5) "예시처럼"이 가장 강력하다 — 잘 된 컴포넌트 1개를 레퍼런스로 지목한다.

### 8.3 스킬로 자동화 (반복 작업)

같은 모양의 작업이 반복되면 스킬로 굳힌다. 스킬 = `SKILL.md`(언제 발동·무엇을) + 선택적 `scripts/`,`references/`. 만들 가치가 있는 후보:

- `bubli-screen-scaffold`: IA 화면 하나를 받아 라우트+컴포넌트+스토리+4상태 골격을 규칙대로 생성. (P0/P1 반복)
- `bubli-glassify`: 컴포넌트를 받아 v20 토큰·유리 질감으로 스타일만 교체. (P2 반복)
- `bubli-api-bind`: feature와 `.http` 엔드포인트를 받아 타입+react-query 훅+로딩/에러 연결. (P3 반복)

만드는 법(요지): 반복 작업의 입력/출력/제약을 `SKILL.md`에 적고, 잘 된 예시 2~3개를 references로 넣고, 발동 설명(description)에 "언제 써야 하는지"를 구체적으로 적는다. 만든 뒤 샘플 2~3개로 돌려보고 결과를 보고 고친다. (스킬 제작 자체를 도와주는 `skill-creator` 스킬이 있으니 그걸 띄워서 만든다.)

### 8.4 병렬 실행 (속도)

서로 안 겹치는 작업은 동시에 돌린다. 안전 규칙: **같은 파일을 동시에 고치지 않는다.** 폴더(feature) 단위로 쪼개면 충돌이 없다.

- 방법 A — 워크트리 격리: 작업마다 별도 git worktree에서 에이전트를 돌리면 서로의 파일을 안 건드린다(작업 끝나면 합침). 디자인 개편처럼 화면별로 갈라지는 일에 적합.
- 방법 B — 서브에이전트 동시 실행: 독립적인 화면 3~4개를 각각 다른 에이전트에 같은 턴에 던진다. 예) 캘린더/메모/알림 화면을 동시에.
- 겹치는 작업(토큰 파일, 공통 ui 컴포넌트)은 병렬 금지. 한 명이 먼저 끝내고 나머지가 그 위에서.

분배 예: P1에서 [대시보드][자료보드][작업판][소통][설정]을 5갈래로 병렬. 공통 `components/ui`는 그 전에 1명이 먼저.

### 8.5 QA 게이트 (매 단계 통과 필수)

- 코드: `lint`/`typecheck`/`build` 그린, `cargo check` 그린.
- 화면: 스토리북에서 4상태 확인, 하드코딩 색 0개.
- 규칙: 이모지 0, 새 색 0, `/app` 밖 업무화면 0.
- 큰 단계 끝엔 서브에이전트로 교차 검수(diff 읽기 + 규칙 위반 점검).

## 10. 결정 사항 (내가 확정함, 추가 질문 없음)

- 디자인 톤: v20 유리·파스텔 유지·발전. 새 톤 재검토 안 함.
