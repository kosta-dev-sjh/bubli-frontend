# 위젯 메모리 런타임 분석 (2026-07-01)

## 결론

Tauri 위젯 메모리 폭증 의심 원인은 버블마다 Next 라우트 WebView를 별도 창으로 띄우는 구조와, 명시 닫기(`close_widget_window`)가 WebView를 해제하지 않고 숨김 상태로 남기던 동작이었다.

이번 수정은 Tauri/로컬 런타임에만 한정했다. 회원앱 UI, 자료보드, WBS/칸반, dashboard는 수정하지 않았다.

## 수정 결과

### 1. planned -> active 승격

`src/lib/tauri/commands.ts`에서 Rust 구현과 adapter 소비가 확인된 14개 명령을 `TAURI_COMMANDS`와 `tauriCommands`로 옮겼다.

- `backup_local_sqlite`
- `check_local_sqlite_integrity`
- `flush_sync_outbox`
- `read_activity_context`
- `recover_timer_state`
- `record_widget_usage_event`
- `restore_local_sqlite_backup`
- `rollup_widget_usage`
- `scan_managed_folder`
- `search_local_files`
- `select_managed_folder`
- `sync_room_messages`
- `sync_widget_usage_summary`
- `watch_managed_folder`

`PLANNED_TAURI_COMMANDS`와 `plannedTauriCommands`는 새 미구현 IPC를 위한 빈 placeholder로 남겼다.

### 2. 명시 닫기 WebView 해제

`close_widget_window`는 더 이상 `MINIMIZED + hide()`로 처리하지 않는다.

- 상태값: `mode=DEFAULT`, `window_visible=false`, `dock_orb_visible=false`, `click_through=false`
- 실제 창: `bubli-widget-{bubbleType}` WebViewWindow를 `destroy()`로 해제

최소화는 그대로 `set_widget_window_mode({ mode: "MINIMIZED" })` 또는 `toggle_widget_window` 흐름을 사용한다. 이 경로는 WebView를 보존한다.

### 3. 앱 종료 시 위젯 창 정리

Tauri 앱 `RunEvent::ExitRequested`와 `RunEvent::Exit`에서 `bubli-widget-*` 라벨 창을 찾아 `destroy()`한다.

이 정리는 열린 창과 숨겨진 창 모두를 대상으로 한다.

### 4. QA 전체 위젯 가드 확인

현재 기본 코드 경로는 Tauri QA 실행 시 전체 8개 버블 창을 동시에 열지 않는다.

- `/app/desktop/widgets?autoOpen=todo`는 한 버블만 연다.
- `/desktop-widget?bubble=todo`는 단일 버블 표면이다.
- Storybook/Visual QA의 8종 표시는 브라우저 캡처용이며 Tauri WebView 창 8개를 동시에 생성하지 않는다.
- `BUBLI_TAURI_WIDGET_QA_ALL=1`이면 Tauri `RunEvent::Ready`에서 8개 버블 창을 QA용으로 한 번에 연다.

따라서 기본 실행은 안전한 단일 앱 기동으로 유지하고, 다중 버블 스트레스 QA는 env를 명시했을 때만 켜지게 했다.

## 검증

아래 명령으로 확인한다.

```bash
npm run typecheck
npm run check:tauri-boundaries
npm run check:product-rules
cargo check --manifest-path src-tauri/Cargo.toml
```

실제 Tauri 창 확인은 가능하면 아래 명령으로 진행한다.

```bash
npm run tauri:dev
```
