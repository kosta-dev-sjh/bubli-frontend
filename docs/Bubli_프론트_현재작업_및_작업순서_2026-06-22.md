# Bubli 프론트 현재 작업과 작업 순서

이 문서는 프론트 담당자가 지금 바로 진행할 수 있는 작업을 정리한다. 기준은 `Bubli_최종기획_완성본_v15_DB회의반영_2026-06-24.md`, 최신 `10_API-Design.md`, 백엔드 개발 가이드, `docs/Bubli_프론트엔드_개발_가이드_2026-06-22.md`, `GIT_CONVENTION.md`다.

백엔드의 세부 DTO가 아직 확정되지 않아도 프론트에서 먼저 만들 수 있는 영역이 있다. 지금은 API 구현을 기다리는 시간이 아니라, 화면 구조와 연결 경계를 잡는 단계다.

## 현재 완료된 기본 틀

| 구분 | 상태 | 위치 |
|---|---|---|
| Next.js 기본 설정 | 완료 | `package.json`, `next.config.ts`, `tsconfig.json` |
| Tauri 기본 설정 | 완료 | `src-tauri/` |
| 공개 사이트 라우트 | 완료 | `/`, `/features`, `/download`, `/faq` |
| 인증 라우트 | 완료 | `/login` |
| 회원 웹 앱 라우트 | 완료 | `/app`, `/app/project-rooms`, `/app/resources`, `/app/project-rooms/[roomId]/work`, `/app/chat`, `/app/settings` |
| Tauri 전용 소통 화면 후보 | 완료 | `/app/desktop/communication` |
| 공통 API client 위치 | 완료 | `src/lib/api` |
| Tauri IPC wrapper 위치 | 완료 | `src/lib/tauri` |
| WebSocket topic 위치 | 완료 | `src/lib/websocket` |
| 디자인 v20 구현 기술 스택 | 완료 | Tailwind v4, TanStack Query, Zustand, dnd kit, TanStack Virtual, Motion, LiveKit, Tauri plugin |

현재 단계는 “기본 틀 없음”이 아니라 “기본 틀 위에 화면과 API 연결을 얹는 단계”다.

## 지금 바로 할 수 있는 일

| 작업 | 할 수 있는 이유 | 산출물 |
|---|---|---|
| 라우트 구조 생성 | v15에서 공개 사이트, 인증, 회원 앱 경로가 확정되어 있음 | `src/app` 라우트 |
| 레이아웃 뼈대 생성 | 화면 역할이 확정되어 있음 | 공개 레이아웃, 회원 앱 레이아웃, Tauri 전용 화면 |
| feature 폴더 생성 | 백엔드 도메인 경계와 프론트 화면 책임이 정리되어 있음 | `src/features/*` |
| 공통 API client 생성 | 공통 응답 형식은 `success/data/error` 기준으로 잡을 수 있음 | `src/lib/api` |
| API 함수 파일 생성 | 최신 `10_API-Design.md`에 경로 기준이 있음 | feature별 `api/*.ts` |
| 임시 DTO 타입 생성 | 백엔드 Entity가 아니라 Response DTO 기준으로 나중에 교체 가능 | `src/types/api`, `src/features/*/types` |
| WebSocket 연결 위치 생성 | topic은 최신 API 설계에 있음 | `src/lib/realtime`, `features/communication` |
| LiveKit 연결 위치 생성 | token은 서버에서 받는 기준이 확정되어 있음 | `features/communication/voice` |
| Tauri IPC wrapper 생성 | IPC 책임은 기획서와 프론트 가이드에 분리되어 있음 | `src/lib/tauri` |
| 빈 상태 UI 작성 | 실제 데이터가 없어도 사용 흐름을 만들 수 있음 | 각 화면의 empty/loading/error 상태 |

## 아직 기다려야 하는 일

| 작업 | 필요한 입력 |
|---|---|
| 실제 인증 API client 완성 | auth API 경로, 토큰 저장 방식, request/response DTO |
| 모든 API 타입 확정 | Swagger/OpenAPI 또는 백엔드 Response DTO |
| 파일 업로드 최종 구현 | multipart 또는 presigned URL 방식 |
| WebSocket payload 처리 | STOMP 인증 방식, 메시지 DTO |
| LiveKit 입장 처리 | token 응답 DTO, 프로젝트룸 멤버 권한 실패 응답 |
| agent job 결과 연결 | job 완료 이벤트와 결과 id 필드 |
| 타이머/위젯 동기화 완성 | idempotency 요청 DTO와 서버 응답 |

## 작업 순서

### 1단계. 레포 정리와 개발 기준 확인

1. `README.md`를 읽고 서비스 구조를 확인한다.
2. `GIT_CONVENTION.md`를 읽고 브랜치, 커밋, PR 규칙을 확인한다.
3. `docs/PROJECT_STRUCTURE.md`로 라우트와 폴더 구조를 확인한다.
4. `docs/Bubli_프론트엔드_개발_가이드_2026-06-22.md`로 feature 책임과 API 경계를 확인한다.

### 2단계. Next.js 기본 구조 생성

완료된 단계다. 이후에는 아래 항목을 유지 보수 기준으로 본다.

1. 라우트가 v15와 최신 API 설계 기준에서 어긋나지 않는지 확인한다.
2. 새 화면을 추가할 때 `/app` 밖에 회원 업무 화면을 만들지 않는다.
3. 공개 사이트, 인증, 회원 앱 레이아웃을 계속 분리한다.
4. 빌드와 타입체크가 깨지지 않는지 확인한다.

### 3단계. 공통 기반 코드 생성

일부 완료된 단계다. 다음 작업은 실제 화면에서 이 기반을 쓰도록 연결하는 것이다.

1. `src/lib/api`의 공통 fetch client를 feature API에서 사용한다.
2. `src/types/api`에 백엔드 Response DTO 기준 타입을 추가한다.
3. `src/lib/websocket`에 STOMP client를 붙인다.
4. `src/lib/tauri`에 실제 IPC command별 wrapper를 추가한다.
5. 디자인 토큰과 Tailwind v4 기준을 공통 UI 컴포넌트로 옮긴다.

### 4단계. 핵심 화면 빈 상태 구현

1. 공개 사이트: `/`, `/features`, `/download`, `/faq`
2. 인증: `/login`
3. 회원 앱: `/app`, `/app/project-rooms`, `/app/resources`
4. 프로젝트룸 작업 화면: `/app/project-rooms/[roomId]/work`
5. 소통 화면: `/app/chat`
6. 설정 화면: `/app/settings`
7. Tauri 소통 전용 화면: `/app/desktop/communication`

### 5단계. feature별 API 파일 생성

1. `features/auth/api`
2. `features/project-room/api`
3. `features/resources/api`
4. `features/todo/api`
5. `features/wbs/api`
6. `features/communication/api`
7. `features/agent/api`
8. `features/widget/api`
9. `features/timer/api`
10. `features/notification/api`
11. `features/calendar/api`
12. `features/managed-folder/api`
13. `features/activity/api`
14. `features/settings/api`

API 함수는 실제 DTO가 확정되기 전까지 얇은 wrapper로만 둔다. 백엔드 Entity를 프론트 타입으로 가정하지 않는다.

### 6단계. 백엔드 계약 수신 후 연결

1. Swagger/OpenAPI 또는 `.http` 예시를 받는다.
2. request/response 타입을 확정한다.
3. feature별 API 함수를 실제 endpoint에 연결한다.
4. 화면의 loading, empty, error, success 상태를 붙인다.
5. WebSocket, LiveKit, Tauri IPC를 실제 흐름에 연결한다.

## 프론트 구현 우선순위

| 우선순위 | 작업 | 이유 |
|---|---|---|
| 1 | Next.js 라우트와 레이아웃 | 모든 화면의 기준점 |
| 2 | 공통 API client와 타입 구조 | 백엔드 계약이 나왔을 때 바로 연결하기 위함 |
| 3 | 공개 사이트와 인증 화면 | 사용자가 처음 만나는 흐름 |
| 4 | `/app` 대시보드, 프로젝트룸, 자료보드 | 회원 앱의 기본 업무 흐름 |
| 5 | WBS/작업판, TODO | Bubli의 핵심 실행 화면 |
| 6 | 소통과 보이스챗 | 웹과 Tauri에서 함께 써야 하는 흐름 |
| 7 | Tauri IPC wrapper | 로컬 폴더, SQLite, 위젯, 타이머 연결 준비 |
| 8 | 버블 위젯 | 웹 화면 복제가 아니라 개인 작업 인터페이스로 구현 |

## 작업 중 판단 기준

- URL은 v15와 `PROJECT_STRUCTURE.md`를 따른다.
- 화면 이름과 feature 이름이 백엔드 패키지 이름과 완전히 같을 필요는 없다.
- API 호출과 데이터 원본 책임은 백엔드 개발 가이드의 도메인 경계를 따른다.
- 프론트와 Tauri는 API 서버만 호출한다.
- 에이전트 모듈이나 agent 컨테이너를 프론트에서 직접 호출하지 않는다.
- Tauri SQLite는 서버 원본이 아니라 캐시, 복구, 로컬 원문, 대기열 역할로 둔다.
- 개인 에이전트 원문과 위젯 상세 사용 이벤트 원문은 서버로 보내지 않는다.

## 완료 기준

- v15 기준 라우트가 모두 존재한다.
- `/app` 밖에 회원 전용 업무 화면이 없다.
- feature별 `api`, `components`, `types` 위치가 잡혀 있다.
- 공통 API 응답 타입이 준비되어 있다.
- 백엔드 DTO가 확정되기 전인 파일에는 임시 타입임을 주석으로 남긴다.
- 실제 API 연결 전에도 화면이 깨지지 않는 empty/loading/error 상태가 있다.

## 검증 명령

현재 프론트 레포에서는 아래 명령을 기준으로 검증한다.

```bash
npm run lint
npm run typecheck
npm run build
cd src-tauri
cargo check
```

`npm run tauri:dev`는 실제 앱 창이 뜨는 명령이라, 화면 확인이 필요할 때 실행한다.
