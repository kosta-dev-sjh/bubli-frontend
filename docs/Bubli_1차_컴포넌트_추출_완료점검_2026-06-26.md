# Bubli 1차 컴포넌트 추출 완료 점검

작성일: 2026-06-26  
기준: v15 기획/API, 디자인보드 v20, 현재 브랜치 `feature/design-board-v20-polish-local`

## 결론

1차 컴포넌트 추출은 **완료 기준을 통과한 상태**로 본다.

지금 필요한 일은 컴포넌트를 더 많이 만드는 것이 아니라, 디자인보드 v20 기준으로 공통 토큰과 기본 컴포넌트를 다시 잡고 기존 화면을 재조립하는 것이다. API 계약이 확정되기 전까지는 화면 구조와 feature별 API client 자리만 유지한다.

## 현재 수치

| 항목 | 현재 상태 | 판단 |
|---|---:|---|
| 실제 page route | 13개 | 공개 사이트, 로그인, 회원 앱, 데스크탑 전용 화면까지 1차 라우트가 있다. |
| feature component | 136개 | v15 핵심 기능을 화면 조각으로 대부분 뽑았다. |
| Storybook story | 144개 | 사용자 화면, 점검용 경계, Tauri 경계, 예외 상태까지 포함한다. |
| feature folder | 18개 | 기획의 주요 기능 묶음과 맞다. |
| feature API folder | 15개 | API 연결 자리는 잡혀 있고, 확정 DTO가 오면 교체하면 된다. |

| 영역 | 라우트 | 상태 |

## 완료 기준

1차 컴포넌트 추출은 아래 기준을 만족하면 완료로 본다.

| 기준 | 현재 판단 |
|---|---|
| v15 핵심 화면이 라우트에 존재한다 | 충족 |
| 프로젝트룸, 자료보드, 대시보드, 작업판, 소통, 설정, 버블이 분리되어 있다 | 충족 |
| 게스트, 이메일/비밀번호 회원가입, 프로젝트/프로젝트룸 분리 같은 삭제된 흐름이 새 라우트 기준에 없다 | 충족 |
| 핵심 기능마다 Storybook에서 확인할 대표 컴포넌트가 있다 | 충족 |
| API 계약이 확정되기 전에도 화면을 확인할 수 있다 | 충족 |
| 디자인보드 v20 전면개편 때 살릴 구조와 버릴 점검용 패널을 구분할 수 있다 | 부분 충족. 계속 triage 문서로 관리 |

## 이제 하지 않을 일

- 1차 컴포넌트 수를 더 늘리기 위해 비슷한 패널을 계속 만들지 않는다.
- 실제 사용자 라우트에 정책, 경계, API 설명 패널을 계속 붙이지 않는다.
- API 계약이 확정되기 전에 화면 전체를 실제 데이터 중심으로 묶지 않는다.
- 디자인보드 v20 정합화 전에 세부 여백과 색을 컴포넌트별로 따로 고치지 않는다.

## 다음 작업

1. Storybook 대표 컴포넌트를 기준으로 공통 디자인 토큰과 기본 컴포넌트를 다시 잡는다.
2. 자료보드, 대시보드, 프로젝트룸, 소통, 버블 순서로 실제 라우트를 디자인보드 v20 기준으로 재조립한다.
3. 백엔드 API 계약이 확정된 기능부터 feature별 `api/`와 `types/`를 교체한다.
4. Tauri 전용 기능은 `/app/desktop/*`, `src/lib/tauri`, `src-tauri` 경계에서만 검증한다.
