# Bubli 프론트·타우리 구현계획 (벤치마킹 반영 고도화)

작성일: 2026-06-26 · 담당: 프론트엔드/타우리 · 레퍼런스: drifty.so, github.com/chrisryugj/Docufinder

이 문서는 외부 레퍼런스 두 개를 우리 기획·DB·RAG 설계에 맞물려, 프론트/타우리 담당이 실제로 무엇을 어떻게 만들지로 바꾼 계획이다. 디자인 개편 계획과는 별개의 "기능 구현" 계획이다.

---

## 0. 한 줄 결론

Docufinder는 우리가 만들려는 "자료보드 + 개인 관리 폴더 + 문서 검색/분석"의 절반을, Tauri+Rust로 이미 제품화한 살아있는 참고작이다. 그대로 베끼는 게 아니라, **무거운 AI(임베딩·RAG·OCR)는 우리 설계대로 서버(pgvector)에 두고, Docufinder에서는 "로컬에서 빠르게 찾고 미리 추출하는 부분"만 가져온다.**

핵심 결정 한 줄: **검색을 2계층으로 나눈다.** 개인 관리 폴더는 Tauri 안에서 로컬로 내용까지 찾고(업로드 없이, 오프라인), 사용자가 올리기로 한 자료만 서버로 보내 의미 검색·RAG를 한다. 이게 우리 프라이버시 원칙("개인 원문은 서버에 안 올림")과도 정확히 맞는다.

주의: Docufinder 라이선스는 BSL 1.1이다. 학습·참고는 자유지만, 코드를 그대로 가져다 상용/배포 제품에 넣는 건 2030년 전까지 제한된다. 우리는 **구조와 패턴을 배우고 직접 다시 구현**한다. 코드 복사는 하지 않는다.

## 1. Docufinder는 무엇이고 왜 보나

Docufinder(앱 이름 "Anything")는 Tauri 2 + React 19 + Rust로 만든 **100% 로컬 문서 내용 검색 엔진**이다. 우리와 같은 기술 스택(Tauri+React+Rust+SQLite FTS5)이라 참고 가치가 매우 높다.

하는 일을 우리 말로 풀면 이렇다.

- 폴더를 등록하면 그 안의 문서를 자동으로 색인한다. 파일명이 아니라 **문서 안 내용**으로 찾는다.
- 지원 형식이 넓다: HWPX/HWP(한글), DOCX, PPTX, XLSX/XLS, PDF(스캔은 OCR), 이미지(OCR), TXT/MD.
- 검색 모드가 네 가지다: 키워드(FTS5), 시맨틱(의미), 하이브리드(둘을 RRF로 합침), 파일명(즉시).
- 파일을 추가/수정/삭제하면 자동 재색인(실시간 동기화).
- 선택 기능으로 AI 질의응답(RAG)과 요약이 있고, 끄면 네트워크 통신이 0이 된다. 원본 파일은 절대 복사하지 않고 색인만 만든다.

기술 구성도 우리와 겹친다: 검색은 SQLite FTS5 + usearch(HNSW 벡터) + RRF 병합, 한국어는 Lindera 형태소 분석, 임베딩은 로컬 ONNX(KoSimCSE 768차원), OCR은 PaddleOCR ONNX, RAG는 Gemini.

한마디로, **우리 자료보드와 개인 관리 폴더가 하려는 일을 데스크탑 제품으로 먼저 만든 사례**다.

## 2. Bubli와 겹치는 지점 (대응표)

우리 AI/RAG 기능 명세(`12_AI기능_RAG_구분`)와 Tauri SQLite 설계(`09C`)를 Docufinder와 맞대보면 이렇다.

| Docufinder 기능 | Bubli의 대응 | 우리 설계 위치 |
|---|---|---|
| 폴더 등록·실시간 감시·재색인 | 개인 관리 폴더 scan/watch | `local_files`, `local_file_events`, Tauri `folder_watch` |
| 문서 텍스트 추출(HWP/PDF/Office) | 자료 텍스트 추출(추출→chunk→embedding) | RAG 1~3단계, 서버 |
| 키워드 검색(FTS5) | 자료보드 키워드 검색 | 서버 자료 + 로컬 FTS5(신규) |
| 시맨틱 검색(usearch) | 자료 의미 검색(17번) | 서버 `resource_embeddings`(pgvector) |
| 관련 문서 추천 | 관련 문서 추천(9번) | 서버 `resource_relations` |
| AI 질의응답(RAG) | 자료 요약·확인질문·요구사항 후보 등 | `agent_suggestions` (후보만) |
| 파일명 즉시 검색 | 자료/파일 빠른 찾기 | 로컬 인메모리/FTS5 |
| OCR | 스캔 PDF/이미지 자료 | 서버(또는 보류) |
| AI 끄면 오프라인 | 개인 원문 비업로드 원칙 | `user_privacy_consents`, 로컬 우선 |

겹침이 이렇게 많다는 건, 우리 기획이 허황된 게 아니라 **이미 누군가 같은 스택으로 만든 현실적 범위**라는 뜻이다.

## 3. 핵심 결정 — 로컬(Tauri) vs 서버(Spring+pgvector) 분담

Docufinder는 전부 로컬이다. 우리는 다르다. 우리는 다중 사용자, 프로젝트룸 공유, 권한, 서버 원본이 있어서 무거운 AI를 서버에 둔다. 그래서 분담을 분명히 정한다.

| 일 | 어디서 | 이유 |
|---|---|---|
| 임베딩 생성·저장 | **서버(pgvector)** | 우리 설계 원본. 420MB 로컬 모델을 4주에 번들하지 않는다. |
| 시맨틱/RAG 검색 | **서버** | 권한 필터(room_members, visibility)가 서버에 있어야 안전 |
| OCR | **서버(또는 보류)** | PaddleOCR 로컬 번들은 무겁다. 4주엔 우선순위 낮음 |
| 폴더 감시·로컬 이벤트 | **로컬(Tauri)** | 사용자 기기의 파일 변화는 Tauri만 본다 |
| 파일명 즉시 검색 | **로컬(Tauri)** | 오프라인·즉시 응답. Docufinder식 인메모리/FTS5 |
| 개인 폴더 "내용" 키워드 검색 | **로컬(Tauri FTS5, 신규)** | 업로드 안 한 개인 파일도 내용으로 찾게. 프라이버시 원칙과 일치 |
| 개인 파일 텍스트 추출(가벼운 형식) | **로컬(Tauri)** | 로컬 FTS5에 넣을 본문 확보. 서버에 원문 안 올림 |

여기서 이 계획의 가장 중요한 추가가 나온다.

### 3.1 검색을 2계층으로 — 이게 핵심 업그레이드

- **로컬 계층 (Tauri, Docufinder식)**: 개인 관리 폴더의 파일을 로컬 SQLite FTS5에 색인한다. 파일명 + 본문 키워드로 즉시, 오프라인으로 찾는다. 원본도 본문도 서버로 보내지 않는다.
- **서버 계층 (Spring + pgvector)**: 사용자가 "올리기/공유"를 켠 자료만 서버로 보내 chunk·embedding·의미검색·RAG·후보 생성을 한다.

이 두 계층을 자료보드 검색창에서 한 번에 보여준다. 로컬 결과가 먼저 즉시 뜨고, 서버 의미검색 결과가 뒤이어 합쳐진다. 이건 Docufinder의 "FTS 먼저 떠서 바로 검색, 벡터는 백그라운드" UX와 같은 방식이다.

왜 좋은가: 프리랜서는 클라이언트 자료가 민감하다. "안 올려도 내 PC에서 내용으로 찾을 수 있다"가 신뢰를 만든다. 우리 기획의 "개인 원문 비업로드" 원칙을 기능으로 증명하는 셈이다.

## 4. 프론트/타우리 담당이 실제로 구현할 것

### 4.1 Tauri 네이티브 모듈 지도 (Drifty + Docufinder 합본)

지금 `src-tauri/src/commands`는 비어 있다. 여기에 아래를 채운다.

```
src-tauri/src/
├── commands/      # 프론트에서 호출하는 IPC 진입점
├── sqlite/        # local_files, local_file_events, 로컬 FTS5 본문 색인, 캐시, outbox
├── folder_watch/  # 폴더 실시간 감시 → local_file_events  (Docufinder식)
├── extract/       # 개인 파일 텍스트 추출(txt/md/docx/pdf 우선)  (Docufinder식, 신규)
├── search/        # 로컬 FTS5 키워드 + 파일명 즉시 검색  (Docufinder식, 신규)
├── activity/      # 활성 앱·창 제목 감지  (Drifty식)
├── window/        # 버블/위젯 창 제어
└── backup/        # 로컬 SQLite 백업·복구
```

`extract`와 `search`가 Docufinder에서 새로 가져오는 부분이고, `activity`가 Drifty에서 가져오는 부분이다.

### 4.2 Rust 텍스트 추출 (extract) — 범위 결정

Docufinder는 hwpx/hwp/docx/pptx/xlsx/pdf/이미지를 다 Rust로 추출한다. 우리는 4주라서 형식을 나눈다.

- **로컬에서 추출(쉬움, 먼저)**: txt, md, docx, pdf(텍스트 레이어 있는 것). 이건 Rust 크레이트로 가능하고 로컬 FTS5에 바로 넣는다.
- **서버에서 추출(어차피 업로드한 자료)**: docx/pptx/xlsx는 Java Apache POI, pdf는 PDFBox. 임베딩 직전에 서버가 추출한다.
- **보류/위험으로 분리**: HWP/HWPX(한글)는 까다롭다. Docufinder는 kordoc(npm)로 푼다. 우리는 4주 안에 "지원 형식 목록에는 두되, 실제 추출은 시연 범위로 제한"한다. 한국 프리랜서에게 HWP는 중요하니, 위험 항목으로 명확히 남긴다.

### 4.3 로컬 검색 (search) — 검색 UX

Docufinder의 4모드를 우리 범위로 줄인다.

- **파일명**: 로컬 인메모리/FTS5, 즉시. (4주 구현)
- **키워드(본문)**: 로컬 FTS5, 개인 폴더 본문. (4주 구현)
- **의미검색**: 서버 pgvector. (4주 구현, 서버팀과 계약)
- **하이브리드(RRF 병합)**: 로컬 키워드 + 서버 의미 결과를 합쳐 정렬. (시연/스트레치)

프론트(자료보드)에서:
- 검색창에 모드 토글(파일명/키워드/의미)을 둔다.
- 로컬 결과는 즉시, 서버 결과는 도착하는 대로 합쳐 보여준다(점진적).
- 결과 카드에 **근거(파일·페이지/시트·행)**와 미리보기를 단다. Docufinder가 "근거 문서 + 페이지"를 보여주듯, 우리도 후보·요약에 근거 chunk를 함께 보여준다(우리 RAG 명세에도 "근거 chunk를 남기라"고 적혀 있다).
- 클릭은 미리보기, 더블클릭은 파일 열기(또는 자료 상세).

### 4.4 한국어 검색 품질 (Lindera)

Docufinder는 한국어를 Lindera 형태소 분석으로 처리한다. 한국어 FTS는 형태소 분석 없이는 "계약서"와 "계약"이 안 걸리는 문제가 생긴다. 우리도 로컬 FTS5에 한국어 토크나이저를 붙여야 검색이 쓸 만해진다. 이건 작지만 빠뜨리면 검색이 "되는 척"만 하게 되는 포인트라 계획에 박아둔다.

### 4.5 폴더 실시간 감시 (folder_watch)

이미 설계(`local_files`, `local_file_events`)가 있다. Docufinder의 패턴(추가/수정/삭제→자동 재색인)을 그대로 따른다. 프론트(개인 관리 폴더 화면)는 색인 진행률과 "최근 변경 N건 반영됨" 상태를 보여준다.

### 4.6 활동 인식 (activity, Drifty)

별도 문서(`04_기술검토/드리프티_벤치마킹...`)에서 정한 대로, 동의 후 활성 앱·창 제목·머문 시간을 `activity_logs`로 기록한다. 분류 축은 집중/딴짓이 아니라 `room_id`(어느 프로젝트 일)다. 이미 DB에 컬럼이 있다.

### 4.7 프론트 feature 매핑

| feature | 이번에 붙는 것 |
|---|---|
| `resources` | 2계층 검색창(모드 토글), 점진적 결과, 근거·미리보기, 의미검색 |
| `managed-folder` | 폴더 등록, 색인 진행률, 실시간 변경 상태 |
| `agent` | 후보(요약/요구사항/WBS/TODO)와 근거 chunk 표시, 승인 흐름 |
| `activity` | 동의 토글, 활동 기반 작업 맥락 제안 |
| `lib/tauri` | search/extract/watch/activity IPC 래퍼 |

## 5. 프라이버시 프레이밍 (랜딩·포트폴리오에 그대로 쓸 무기)

Docufinder가 "AI 끄면 통신 0, 원본 복사 안 함, 색인만"을 강조하는 건 우리에게도 좋은 메시지다. 우리 버전:

- 개인 관리 폴더는 **내 PC 안에서만** 색인·검색한다. 원본도 본문도 서버로 안 올라간다.
- 서버로 가는 건 사용자가 직접 "올리기/공유"를 켠 자료뿐이다.
- 활동 인식은 동의가 먼저고, 화면 녹화·키 입력은 안 한다.

이 세 줄은 프리랜서의 "클라이언트 자료 민감함" 불안을 정확히 건드린다. 랜딩 카피와 설정 화면 문구로 그대로 쓴다.

## 6. 4주 범위 / 시연 vs 구현 / 위험

이번 4주에 실제로 만들 것:
- 폴더 감시 + 로컬 FTS5(파일명 + 본문 키워드, txt/md/docx/pdf), 색인 진행률 UI
- 서버 의미검색(pgvector) 연결, 자료보드 2계층 검색창
- 활동 인식(동의 + 기록 + 작업 맥락 제안 1차)
- 근거 chunk를 보여주는 후보/요약 UI

시연만(구현은 얇게):
- 하이브리드 RRF 병합, HWP 로컬 추출, OCR, 로컬 임베딩

검증 위험으로 분리:
- HWP/HWPX 추출(한국 프리랜서 필수인데 어렵다)
- 대용량 폴더 색인 성능·발열 (Docufinder도 1000건 2~5분)
- 한국어 형태소 토크나이저 없으면 검색 품질 저하
- Mac 권한(폴더 접근, 활동 인식 접근성)
- 로컬/서버 결과 병합 시 권한 필터 누락 금지

## 7. 검증 방법

- 로컬 검색: 내 폴더에 샘플 문서를 넣고, 파일명·키워드로 즉시 결과가 뜨는지.
- 한국어: "계약/계약서/계약금" 같은 변형이 걸리는지 형태소 토크나이저로 확인.
- 2계층 병합: 로컬 결과가 먼저, 서버 의미 결과가 뒤이어 합쳐지는지.
- 근거 표시: 요약·후보에 출처 파일·페이지가 같이 보이는지.
- 프라이버시: 개인 폴더 검색 시 네트워크 요청이 안 나가는지(개발자도구/패킷으로 확인).
- 팀 검증: 팀원 PC에서 폴더 색인 1회 돌려 성능·정확도 메모로 남김.

1. 로컬 본문 추출 형식 범위 확정: txt/md/docx/pdf까지 4주에 넣을지, docx/pdf는 서버로 미룰지.
2. 한국어 토크나이저 선택: 로컬 FTS5에 Lindera(Rust)를 붙일지, 단순 bigram으로 갈지.
3. 2계층 검색 UI: 모드 토글을 노출할지, "그냥 검색" 한 칸으로 합치고 내부에서 병합할지.
4. HWP 지원을 시연 목록에 넣을지(넣되 위험으로 표시 권장).
5. OCR은 이번 범위에서 제외 확정할지.

## 부록. 두 레퍼런스 한 줄 정리

- drifty.so → 활동 인식(활성 앱·창·시간). 우리는 집중/딴짓이 아니라 "어느 프로젝트 일"로 묶는다. DB에 `activity_logs.room_id`로 이미 반영.
- Docufinder → 로컬 문서 내용 검색(Tauri+Rust+FTS5+벡터+RAG). 우리는 무거운 AI는 서버에 두고, "로컬 즉시 검색 + 개인 폴더 본문 검색 + 폴더 감시 + 가벼운 추출"만 가져온다. 라이선스(BSL 1.1)상 코드 복사 없이 패턴만 차용한다.
