# 프론트 명세 분리 기준

작성일: 2026-06-29

이 문서는 Bubli 프론트 작업에서 어떤 판단을 어느 문서에서 봐야 하는지 정리한다.

문제가 생긴 이유는 디자인보드, 기획서, 라우트 명세, Tauri 명세가 한 덩어리처럼 보였기 때문이다. 이제 문서마다 맡는 책임을 분리한다.

## 결론

프론트 구현 기준은 아래 순서로 본다.

```text
기획 최종본 -> IA -> 와이어프레임 -> 라우트 명세 -> 화면 구현 순서 -> Visual QA
```

디자인보드는 이 순서에 끼지 않는다. 디자인보드는 마지막에 색, 질감, 밀도, 움직임 방향을 확인할 때 본다.

## 문서별 책임

| 문서 | 답하는 질문 | 고치면 안 되는 것 |
|---|---|---|
| 기획 최종본 v15 | Bubli가 왜 필요한가, 어떤 기능을 만들 것인가 | 화면 배치 세부값 |
| API Design | 화면이 어떤 서버 데이터를 받을 것인가 | 화면 디자인 |
| Data Model | 데이터 원본과 관계가 무엇인가 | 컴포넌트 배치 |
| IA 문서 | 공개 사이트, 회원 앱, Tauri 앱이 어떻게 나뉘는가 | API 필드 상세 |
| 와이어프레임 문서 | 한 화면 안에서 무엇을 어디에 놓을 것인가 | 색, 그림자, 애니메이션 |
| 라우트 명세 | Next.js 라우트와 컴포넌트 책임은 무엇인가 | 사업 방향 변경 |
| 화면 구현 순서 | 어떤 순서로 구현하고 검증할 것인가 | 제품 범위 변경 |
| Tauri 명세 | 데스크탑 앱과 로컬 기능은 무엇인가 | 일반 웹 라우트 구조 |
| 디자인보드 | 어떤 톤으로 보일 것인가 | IA, 라우트, 데이터 흐름 |
| Visual QA 리포트 | 실제 캡처가 기준에 맞는가 | 기획 의사결정 |

## 판단이 갈릴 때의 기준

| 헷갈리는 상황 | 우선 기준 |
|---|---|
| 어떤 화면을 만들어야 하는지 모르겠음 | IA 문서 |
| 라우트가 어디인지 모르겠음 | 라우트 명세 |
| 화면 안에 무엇을 놓을지 모르겠음 | 와이어프레임 |
| 어떤 데이터를 써야 하는지 모르겠음 | API Design, Data Model |
| 이게 웹인지 Tauri인지 모르겠음 | Tauri 명세 |
| 디자인보드와 와이어프레임이 다름 | 와이어프레임 |
| 디자인보드와 기획서가 다름 | 기획 최종본 |
| 구현 화면과 캡처가 다름 | 캡처를 보고 수정 |

## 구현 산출물과 문서 연결

| 구현 산출물 | 기준 문서 | 검증 문서 |
|---|---|---|
| `/app` 대시보드 | IA, 와이어프레임, 라우트 명세 | Visual QA 리포트 |
| `/app/resources` 자료보드 | IA, 와이어프레임, API Design | Visual QA 리포트 |
| `/app/project-rooms/{roomId}` 프로젝트룸 홈 | IA, 와이어프레임, Data Model | Visual QA 리포트 |
| `/app/project-rooms/{roomId}/work` 작업판 | 와이어프레임, API Design, Data Model | Visual QA 리포트 |
| `/app/desktop/widgets` 위젯 미리보기 | Tauri 명세, 라우트 명세 | Visual QA 리포트 |
| Tauri 위젯 창 | Tauri 명세 | 데스크탑 실행 확인 |
| Storybook 컴포넌트 | 라우트 명세, UI 토큰 | Storybook 빌드, Visual QA |

## 디자인보드 사용 규칙

디자인보드에서 가져올 수 있는 것:

- 색의 느낌
- 유리감
- 표면 밀도
- 버블의 형태감
- 라이트와 다크의 분위기 차이
- 버튼, 칩, 카드의 촉감

디자인보드에서 가져오면 안 되는 것:

- 실제 사이트 구조
- 실제 메뉴 구조
- 실제 라우트
- 실제 데이터 흐름
- 프로젝트룸과 대시보드의 책임
- Tauri 앱과 웹 앱의 경계
- 존재하지 않는 기능

## 기능 하나를 명세로 쪼개는 방법

예를 들어 `자료보드`를 구현한다면 아래처럼 나눈다.

| 명세 영역 | 적을 내용 |
|---|---|
| 기획 | 자료가 흩어져 업무 기준을 잃는 문제를 푼다 |
| IA | 개인 자료보드와 프로젝트룸 자료보드를 분리한다 |
| 와이어프레임 | 업로드, 분석 상태, 공유 후보, 확인 필요 항목을 배치한다 |
| API | resources, agent jobs, agent suggestions를 연결한다 |
| 데이터 | `resources`, `agent_jobs`, `agent_suggestions`를 쓴다 |
| 라우트 | `/app/resources`, `/app/project-rooms/{roomId}/resources` |
| 컴포넌트 | ResourceFlowView, resource cards, suggestion cards |
| Visual QA | light, dark 캡처를 남긴다 |

이렇게 나누면 디자인보드가 화면 구조를 대신하지 못한다.

## 프론트 작업 요청 템플릿

프론트 담당에게 작업을 줄 때는 아래 형식으로 준다.

```text
작업명:
대상 라우트:
기준 문서:
사용 컴포넌트:
연결 데이터:
Tauri 필요 여부:
디자인 참고:
완료 기준:
금지 사항:
```

예시:

```text
작업명: 개인 자료보드 화면 연결
대상 라우트: /app/resources
기준 문서: 02_회원웹앱_와이어프레임, 04_프론트_구현_라우트명세
사용 컴포넌트: ResourceFlowView
연결 데이터: resources API, agent jobs API
Tauri 필요 여부: 없음
디자인 참고: Sky Opal 표면, Night Bubble 다크 톤
완료 기준: route-resources-light/dark 캡처 생성, typecheck 통과, 금지색 0건
금지 사항: 디자인보드 HTML 복사, 에이전트 직접 호출, 링크 초대 추가
```

## 명세 수정 원칙

문서를 고칠 때는 아래 순서를 따른다.

| 바뀐 내용 | 같이 확인할 문서 |
|---|---|
| 새 라우트 추가 | IA, 와이어프레임, 라우트 명세, Visual QA 대상 |
| 새 API 추가 | API Design, 라우트 명세, 구현 순서 |
| 새 DB 필드 추가 | Data Model, API Design, 화면 명세 |
| Tauri 기능 추가 | Tauri 명세, 라우트 명세, 보안 기준 |
| 디자인 톤 변경 | 디자인 토큰, Visual QA 기준 |
| 기능 제외 | 기획 최종본, 라우트 명세, 금지 목록 |

한 문서만 고치고 끝내면 다음 작업자가 다시 헷갈린다.

## 완료 기준

프론트 명세가 제대로 분리됐다고 보려면 아래가 맞아야 한다.

| 기준 | 설명 |
|---|---|
| 목적 분리 | 기획, 화면, 라우트, API, Tauri, 디자인의 책임이 다르다 |
| 참조 순서 | 프론트 담당이 무엇을 먼저 볼지 안다 |
| 금지 기능 차단 | v15에서 빠진 기능이 다시 생기지 않는다 |
| 디자인보드 위치 | 화면 구조가 아니라 디자인 참고로만 쓰인다 |
| 검증 방법 | Visual QA와 typecheck 기준이 있다 |

한마디로, 명세 분리는 "무엇을 만들지"와 "어떻게 보이게 할지"를 분리하는 일이다. Bubli는 기획 흐름이 먼저고, 디자인보드는 그 흐름을 보기 좋게 만드는 참고 자료다.
