# Bubli 화면설계와 프론트 명세

작성일: 2026-06-29 · 현황 갱신: 2026-06-29

이 폴더는 프론트 구현자가 디자인보드를 그대로 옮기지 않도록, v15 기획서를 화면과 라우트 기준으로 풀어낸 작업 기준이다.

## 30초 현황 요약 (새 사람 먼저 읽기)

지금 프로젝트가 어디까지 왔는지 한눈에 본다. 상태 표기는 `[완료] / [진행] / [예정] / [불일치]`다. (이모지를 쓰지 않는 것은 프로젝트 규칙이다.)

| 영역 | 현황 |
|---|---|
| 화면 | 회원앱 핵심 라우트 일부만 새 화면으로 구현됨(대시보드·개인 자료보드·위젯 미리보기, 모두 flag 뒤·mock). 프로젝트룸 상세 흐름(룸 홈·룸 자료·룸 채팅)은 아직 비어 있고, 일부 라우트는 spec과 이름이 다르다 |
| 컴포넌트 | `src/components` UI Kit 기반은 상당히 구축됨. `src/features/*` feature 화면 정합은 진행 중 |
| Storybook | 168개 스토리로 충실(상태·다크 포함) |
| API | API 래퍼와 호출 구조는 있으나, 주요 화면은 아직 mock/default 데이터 중심이라 실데이터 연결은 거의 없음 |
| Widget | `/app/desktop/widgets`는 Tauri 위젯 설정/상태 검토 화면이다. 실제 데스크탑 위젯 창은 Tauri window로 따로 만든다 |
| Tauri | `src/lib/tauri` 스캐폴딩(is-tauri·ipc·commands)만 있고 실제 연결은 없음 |

라우트별 상태와 코드 ↔ spec 라우트 매핑표는 `04_프론트_구현_라우트명세_2026-06-29.md`의 "현재 코드 라우트 ↔ 목표 spec 라우트 매핑"을 본다.

### 헷갈리지 않기 위한 3가지 경고

1. 디자인보드(`docs/visual-target/`, 디자인 시안)는 구현 대상이 아니다. 색, 질감, 밀도, 유리감, 버블감, 타이포, 그림자, 간격의 시각 참고로만 쓴다. 화면 구조·라우트·데이터 흐름은 항상 이 폴더의 문서를 따른다.
2. `docs/visual-qa/Visual_Delta_Report_2026-06-26.md`는 과거 스프린트(데스크탑 캔버스 + 큰 floating 위젯) 기준이며, 현재 `docs/Bubli_Visual_Target_Guide_2026-06-26.md`보다 우선하지 않는다. 그 방향(큰 장식 버블, 캔버스 위젯)은 폐기됐다.
3. 시각 판단 우선순위: `docs/visual-target/` 이미지 > Visual Target Guide > Brand Bible > Interaction Bible > UI Kit 문서 > 코드.

## 핵심 원칙

디자인보드는 화면 명세서가 아니다.

디자인보드는 색, 밀도, 유리감, 버블감, 움직임의 방향을 보는 참고 자료다. 구현할 화면의 정보 구조, 라우트, 버튼, 입력, 데이터 흐름은 이 폴더의 문서를 따른다.

## 프론트 구현자가 지켜야 할 선

- 공개 사이트와 회원 웹 앱은 분리한다.
- 회원 웹 앱은 프로젝트룸, 자료보드, WBS/작업판, 소통, 일정, 설정을 맡는다.
- 대시보드는 사용자 기준 화면이다. 특정 프로젝트룸 한 개의 상세 화면이 아니다.
- 위젯은 개인 데스크탑 영역이다. 디자인보드 전체를 위젯 안에 복사하지 않는다.
- Tauri 위젯은 20v 디자인보드의 기능 구성을 기준으로 한다. 20v는 시각 참고만이 아니라 기능 체크리스트다.
- 현재 코드나 화면에서 빠져 보이는 Tauri 위젯 기능은 삭제가 아니라 누락 항목으로 보고 추가한다.
- 나머지 화면과 세부 디자인은 단계적으로 손보더라도 Tauri 위젯 기능 범위는 20v보다 작아지면 안 된다.
- 즉, Tauri 위젯은 20v 기능 동등성을 먼저 맞추고 나머지 화면은 이후 단계에서 천천히 손본다.
- Tauri 앱은 회원 웹 앱을 WebView로 띄우고, 그 위에 로컬 기능과 버블 위젯을 붙인다.
- 프론트와 Tauri는 에이전트 서버를 직접 호출하지 않는다. 항상 API 서버를 호출한다.
- 링크 초대, 이메일 주소 초대, 비회원 게스트 초대, `/signup`은 만들지 않는다.

한마디로, 프론트는 "예쁜 보드"를 만드는 것이 아니라 "프리랜서가 받은 자료를 오늘 할 일로 바꾸는 업무 흐름"을 만든다.
