## 결론

Bubli의 프론트는 디자인보드 복제가 아니다.

프론트가 만들어야 하는 것은 아래 흐름이다.

```text
공개 사이트
-> 구글 로그인
-> 회원 웹 앱
-> 프로젝트룸에 자료 업로드
-> 에이전트 후보 확인
-> WBS/TODO/일정 반영
-> 대시보드와 데스크탑 버블에서 다시 확인
```

## 만들 화면과 만들지 않을 화면

| 구분 | 만든다 | 만들지 않는다 |
|---|---|---|
| 로그인 | 구글 로그인 진입 | 이메일/비밀번호 회원가입, `/signup` |
| 공개 사이트 | 소개, 기능 안내, 다운로드, FAQ | 공개 프로필, 매칭, 의뢰인 화면 |
| 프로젝트룸 | 자료, WBS/TODO, 채팅, 일정, 설정 | 프로젝트와 프로젝트룸을 따로 나눈 구조 |
| 초대 | 수락된 친구 선택 초대 | 링크 초대, 이메일 주소 초대, 비회원 게스트 |
| 대시보드 | 내 TODO, 일정, 확인 필요 항목, 진행 중 타이머 | 프로젝트룸 하나만 보는 공용 대시보드 |
| 위젯 | 개인 데스크탑 버블 | 디자인보드 전체를 작은 화면에 그대로 축소 |
| 에이전트 | 후보 생성과 사용자 승인 흐름 | 자동 확정, 자동 DB 반영 |

6. 디자인보드는 마지막에 색, 간격, 유리감, 버블 느낌만 참고한다.

## 디자인보드를 볼 때 주의할 점

디자인보드는 제품 방향을 보여주는 시각 자료다.

아래는 디자인보드에서 가져와도 된다.

- 워터블루, 페일스카이, 라일락, 펄, 레인그레이 계열의 부드러운 톤
- 유리감 있는 패널
- 버블 위젯의 가벼운 떠 있는 느낌
- 라이트/다크 테마의 대비

아래는 그대로 가져오면 안 된다.

- 한 화면에 모든 기능을 몰아넣는 구성
- 실제 라우트와 상관없는 데모 카드
- 프로젝트룸 화면을 대시보드처럼 보이게 만드는 구성
- 위젯을 웹 앱 전체의 축소판처럼 만드는 구성
- 게스트, 링크 초대, 이메일 초대 흐름

## 프론트 완료 기준

화면은 아래 네 가지가 맞아야 완료로 본다.

| 기준 | 확인 방법 |
|---|---|
| 기획 정합성 | v15에서 빠진 기능이 다시 들어오지 않았는지 확인 |
| 라우트 정합성 | 공개 사이트, 회원 앱, Tauri 전용 기능이 섞이지 않았는지 확인 |
| 데이터 정합성 | 화면에 필요한 데이터가 API 또는 Tauri IPC 기준과 연결되는지 확인 |
| 시각 정합성 | Storybook 또는 Visual QA 캡처로 라이트/다크 화면 확인 |

한마디로, 예쁘게 만든 화면보다 "어디에서 무엇을 하는지 헷갈리지 않는 화면"이 먼저다.
