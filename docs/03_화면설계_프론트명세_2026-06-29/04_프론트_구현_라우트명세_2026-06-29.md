# 프론트 구현 라우트 명세

이 문서는 Next.js 프론트가 어떤 라우트를 어떤 책임으로 구현해야 하는지 정리한다.

## 현재 코드 라우트 ↔ 목표 spec 라우트 매핑 (2026-06-29 점검)

이 표는 "지금 코드에 있는 라우트"와 "이 문서가 목표로 하는 라우트"가 어디서 어긋나는지 한눈에 보기 위한 것이다.

- 최종 진실은 이 문서(spec)의 라우트다.
- 단, 라우트 이름과 구조를 실제로 바꾸는 작업은 별도 Phase에서 승인받아 진행한다. 이 표는 현황 기록일 뿐, 코드 변경 지시가 아니다.

상태 표기:

- `[완료]` 새 화면이 구현되어 flag 뒤에 붙어 있음(데이터는 아직 mock/default 중심)
- `[진행]` 화면 파일은 있으나 스캐폴드 단계이거나 spec 정합이 진행 중
- `[예정]` 라우트/화면이 아직 없음
- `[불일치]` 코드 라우트가 spec과 이름 또는 위치가 다름

### 다음 Phase에서 결정할 라우트 정합 항목

아래는 이번 Phase에서 바꾸지 않는다. 별도 승인 후 진행한다.

- `/app/calendar` → `/app/schedule` 정합 여부
- `/app/agent-suggestions` → `/app/agent` 정합 여부
- `/app/chat` → `/app/project-rooms/{roomId}/chat` 이동 여부
- `/app/desktop/communication`의 spec 편입 또는 제거
- `/app/project-rooms/{roomId}` 홈, `{roomId}/resources` 신규 구현

## 라우트 원칙

- 공개 사이트와 회원 웹 앱을 분리한다.
- 회원 웹 앱은 `/app` 아래에 둔다.
- Tauri 앱은 `/app`을 WebView로 열고, 로컬 기능은 Tauri IPC로 붙인다.
- 디자인보드 HTML을 라우트로 그대로 붙이지 않는다.
- Storybook은 컴포넌트 검증용이고, 실제 서비스 라우트는 이 문서를 따른다.

## 공개 사이트 라우트

| 라우트 | 화면 목적 | 주요 블록 | API 필요 |
|---|---|---|---|
| `/` | 서비스 첫 인상 | 문제, 해결 흐름, 주요 화면, 다운로드 CTA | 없음 |
| `/features` | 기능 설명 | 프로젝트룸, 자료보드, 대시보드, 버블, 에이전트 | 없음 |
| `/download` | 앱 다운로드 | macOS, Windows 안내, 설치 주의 | 없음 |
| `/faq` | 자주 묻는 질문 | 권한, 파일, 보안, AI, Tauri | 없음 |
| `/login` | 구글 로그인 진입 | 구글 로그인 버튼, 공개 사이트로 돌아가기 | auth API |

공개 사이트에는 로그인한 사용자의 업무 데이터를 보여주지 않는다.

## 회원 웹 앱 라우트

| 라우트 | 화면 | 책임 | 주요 데이터 |
|---|---|---|---|
| `/app` | 대시보드 | 사용자 기준 오늘 업무 요약 | 내 TODO, 일정, 확인 필요 항목, 타이머, 알림 |
| `/app/project-rooms` | 프로젝트룸 목록 | 업무 단위 선택과 생성 | 내 프로젝트룸, 멤버 상태, 최근 활동 |
| `/app/project-rooms/{roomId}` | 프로젝트룸 홈 | 룸 단위 현황 요약 | 자료, WBS/TODO, 채팅, 멤버, 에이전트 제안 |
| `/app/project-rooms/{roomId}/resources` | 프로젝트룸 자료보드 | 룸 공유 자료 관리 | ROOM_SHARED resources, 댓글, 버전 |
| `/app/project-rooms/{roomId}/work` | WBS/작업판 | 확정 작업 관리 | wbs_items, tasks, schedules |
| `/app/project-rooms/{roomId}/chat` | 프로젝트룸 채팅 | 룸 멤버 소통 | chat_messages, voice rooms |
| `/app/resources` | 개인 자료보드 | 개인 자료와 공유 승인 | PERSONAL resources, 분석 결과, 공유 후보 |
| `/app/schedule` | 일정 | 내 일정과 프로젝트룸 일정 확인 | schedules, calendar sync |
| `/app/agent` | 에이전트 제안함 | 후보 검토 | agent_jobs, agent_suggestions |
| `/app/settings` | 설정 | 프로필, 알림, 언어, 개인정보 | users, preferences, consents |
| `/app/desktop/widgets` | Tauri 위젯 설정/상태 | 실제 데스크탑 버블의 기능, 상태, 권한, 표시 슬롯 검토 | Tauri planned IPC, widget display cache |

## 기능 flag 기준

기능 flag는 새 화면을 기존 라우트에 안전하게 붙일 때만 쓴다.

| flag | 대상 | 설명 |
|---|---|---|
| `NEXT_PUBLIC_BUBLI_NEW_DASHBOARD` | `/app` | 새 대시보드 컴포넌트 적용 |
| `NEXT_PUBLIC_BUBLI_NEW_RESOURCES` | `/app/resources` | 새 ResourceFlowView 적용 |
| `NEXT_PUBLIC_BUBLI_NEW_WIDGET_PREVIEW` | `/app/desktop/widgets` | 새 WidgetPreview 적용 |

flag는 임시 전환 장치다. 화면 기준이 확정되면 라우트 명세와 코드에서 같은 이름으로 정리한다.

## 컴포넌트 책임

| 컴포넌트 계층 | 책임 | 금지 |
|---|---|---|
| `src/components/ui` | 버튼, 칩, 패널, 뱃지 같은 순수 UI | API 호출, Tauri 호출 |
| `src/components/domain` | 자료, 작업, 제안 같은 도메인 카드 | 라우트 전용 상태 과다 포함 |
| `src/components/dashboard` | 대시보드 카드와 배치 | 프로젝트룸 상세 화면으로 변질 |
| `src/components/widget` | 위젯 미리보기와 상태 표현 | 실제 Tauri 권한 처리 |
| `src/features/*` | API, 상태, 화면 조립 | 디자인보드 데모 복사 |
| `src/lib/tauri` | Tauri 런타임 감지와 IPC | 일반 컴포넌트에서 직접 `@tauri-apps/*` 호출 |

## 화면별 데이터 연결

| 화면 | API 또는 IPC | 비고 |
|---|---|---|
| 대시보드 | `/api/me/project-rooms`, TODO, schedules, notifications | 사용자 기준으로 합친다 |
| 자료보드 | resources API, agent jobs API | 개인/프로젝트룸 범위 분리 |
| WBS/작업판 | wbs, tasks, schedules API | 후보 승인 후 확정 데이터 표시 |
| 채팅 | chat API, WebSocket, voice API | Tauri는 local_room_message_cache를 캐시로만 사용 |
| Tauri 위젯 설정/상태 | Tauri planned IPC, widget display cache | 실제 떠 있는 버블 창은 Tauri window로 처리 |
| 개인 관리 폴더 | Tauri IPC, local-file sync API | 사용자가 선택한 폴더만 |

Tauri 위젯은 20v 디자인보드에 있던 기능 구성을 기준으로 한다.
20v는 단순 디자인 참고가 아니라 기능 체크리스트다.
`/app/desktop/widgets`는 그 기능을 검토하는 회원앱 화면이고, 실제 버블 창은 Tauri 쪽에서 만든다.
현재 빠져 보이는 위젯 기능은 삭제된 것이 아니라 다음 구현에서 채울 항목으로 본다.
단, 기획에서 삭제된 받을돈/페이 위젯은 되살리지 않는다.

## Visual QA 기준

라우트에 새 화면을 붙이면 아래를 확인한다.

```bash
cd "04_개발_작업공간/repos/bubli-frontend"
npm run visual:qa
```

빠른 재실행은 아래처럼 한다.

```bash
SKIP_BUILD=1 npm run visual:qa
```

완료 기준은 아래와 같다.

| 기준 | 값 |
|---|---|
| route screenshot | light와 dark 둘 다 생성 |
| md5 | light와 dark가 서로 다름 |
| typecheck | 통과 |
| 금지색 grep | 0건 |
| 캡처 실패 | 없음 |

한마디로, 프론트 구현은 라우트, 데이터, 권한, 캡처가 맞아야 끝난다.
