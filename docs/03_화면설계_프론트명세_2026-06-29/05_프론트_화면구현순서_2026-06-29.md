# 프론트 화면 구현 순서

작성일: 2026-06-29

이 문서는 프론트 담당자가 Bubli 화면을 어떤 순서로 구현해야 하는지 정리한 작업 순서표다.

디자인보드를 먼저 보고 화면을 따라 만들면 서비스 구조가 흐려진다. 먼저 기획 흐름, 라우트, 데이터 책임을 잡고 그 위에 디자인 톤을 얹는다.

## 전체 구현 순서

### 0단계. 기준 잠금

작업을 시작하기 전에 이번 화면이 아래 중 어디에 속하는지 먼저 정한다.

| 구분 | 기준 |
|---|---|
| 공개 사이트 | 로그인 전 방문자가 Bubli를 이해하는 화면 |
| 회원 웹 앱 | 로그인한 사용자가 프로젝트룸, 자료, 할 일, 일정을 관리하는 화면 |
| Tauri 앱 | 회원 웹 앱을 데스크탑에서 열고 로컬 기능을 붙이는 앱 |
| 데스크탑 버블 | 개인 위젯 상태를 작게 보여주는 데스크탑 화면 |

헷갈리면 먼저 `/app` 안인지 아닌지를 본다. 로그인한 사용자의 업무 데이터가 있으면 회원 웹 앱이다.

### 1단계. 라우트부터 잡기

화면을 그리기 전에 라우트와 화면 책임을 먼저 적는다.

| 확인 항목 | 예시 |
|---|---|
| 라우트 | `/app/resources` |
| 화면 책임 | 개인 자료보드 |
| 사용 데이터 | `PERSONAL resources`, 분석 결과, 공유 후보 |
| 연결 API | resources API, agent jobs API |
| Visual QA 대상 | `route-resources-light.png`, `route-resources-dark.png` |

라우트 책임이 애매하면 구현하지 않는다. 먼저 라우트 명세를 고친다.

### 2단계. 데이터 흐름 확인

화면에 들어갈 정보가 어디서 오는지 확인한다.

| 데이터 종류 | 원본 |
|---|---|
| 프로젝트룸, 멤버, 자료, 작업, 일정, 채팅 | 서버 API |
| 에이전트가 만든 후보 | 서버 API의 `agent_jobs`, `agent_suggestions` |
| 데스크탑 위젯 표시 캐시 | Tauri SQLite 캐시 |
| 로컬 폴더, 로컬 타이머 복구 | Tauri IPC |
| 색, 그림자, 반경 | 디자인 토큰 |

프론트와 Tauri는 에이전트 서버를 직접 호출하지 않는다. 에이전트 작업도 API 서버를 거친다.

### 3단계. 낮은 충실도 화면부터 만들기

처음부터 디자인보드처럼 꾸미지 않는다.

먼저 아래만 맞춘다.

- 화면 제목
- 핵심 영역 2개에서 4개
- 주요 CTA
- 빈 상태
- 로딩 상태
- 오류 상태
- 라이트 모드와 다크 모드의 기본 대비

이 단계에서는 장식보다 사용자가 어디를 봐야 하는지가 더 중요하다.

### 4단계. 컴포넌트 책임 분리

컴포넌트는 역할별로 나눈다.

| 위치 | 넣을 것 | 넣지 않을 것 |
|---|---|---|
| `src/components/ui` | 버튼, 칩, 패널, 뱃지 | API 호출, Tauri 호출 |
| `src/components/domain` | 자료 카드, 작업 카드, 후보 카드 | 라우트 전용 상태 |
| `src/components/dashboard` | 대시보드 카드와 배치 | 프로젝트룸 상세 전용 로직 |
| `src/components/widget` | 위젯 상태 표현 | 실제 OS 권한 처리 |
| `src/features/*` | 데이터 요청, 화면 조립 | 디자인보드 HTML 복사 |
| `src/lib/tauri` | Tauri 감지, IPC 래퍼 | 일반 화면 직접 호출 |

한 화면에서만 쓰는 배치는 라우트 가까이에 둔다. 여러 화면에서 반복되는 조각만 공용 컴포넌트로 올린다.

### 5단계. 상태를 빠짐없이 만든다

각 라우트는 최소 아래 상태를 가진다.

| 상태 | 확인 기준 |
|---|---|
| 기본 | 정상 데이터가 있을 때 정보가 읽힌다 |
| 빈 상태 | 아직 데이터가 없을 때 다음 행동이 보인다 |
| 로딩 | 화면이 덜컥이지 않는다 |
| 오류 | 다시 시도하거나 돌아갈 길이 있다 |
| 권한 없음 | 프로젝트룸 접근 불가를 설명한다 |
| 다크 모드 | 같은 정보 구조가 유지된다 |

데이터가 없어도 화면이 무너지면 구현 완료가 아니다.

### 6단계. 디자인 톤 적용

구조가 맞은 뒤에 디자인보드의 톤을 적용한다.

적용할 것:

- Sky Opal 계열의 밝은 표면
- Night Bubble 다크 표면
- 부드러운 유리감
- 버블 위젯의 가벼운 떠 있음
- 라이트와 다크에서 같은 정보 위계

적용하지 않을 것:

- 디자인보드 전체 레이아웃 복사
- 화면 목적과 맞지 않는 큰 장식
- 업무 화면을 랜딩 페이지처럼 구성
- 모든 라우트를 같은 보드 모양으로 통일

### 7단계. Tauri와 데스크탑 기능 분리

웹 앱과 데스크탑 앱을 섞지 않는다.

| 기능 | 구현 위치 |
|---|---|
| 프로젝트룸, 자료보드, 작업판, 채팅, 일정 | 회원 웹 앱 |
| WebView로 `/app` 열기 | Tauri 앱 |
| 로컬 폴더 선택 | Tauri IPC |
| 로컬 SQLite 캐시 | Tauri 앱 |
| 데스크탑 버블 창 | Tauri 앱 |
| 위젯 상태 미리보기 | 웹 앱 `/app/desktop/widgets` |

`/app/desktop/widgets`는 데스크탑 위젯을 검토하는 화면이다. 실제 OS 창, 트레이, 파일 권한은 Tauri 쪽 명세를 따른다.

### 8단계. Visual QA로 확인

라우트 화면을 붙인 뒤에는 캡처로 확인한다.

```bash
cd "04_개발_작업공간/repos/bubli-frontend"
SKIP_BUILD=1 npm run visual:qa
```

완료 기준:

| 항목 | 기준 |
|---|---|
| 라이트 캡처 | 생성됨 |
| 다크 캡처 | 생성됨 |
| md5 | 라이트와 다크가 다름 |
| typecheck | 통과 |
| 금지색 grep | 0건 |
| 화면 겹침 | 없음 |

스크린샷이 이상하면 코드 설명보다 스크린샷을 기준으로 고친다.

### 9단계. 인수인계 남기기

작업이 끝나면 아래를 남긴다.

| 항목 | 적을 내용 |
|---|---|
| 적용 라우트 | 예: `/app/resources` |
| 연결 flag | 예: `NEXT_PUBLIC_BUBLI_NEW_RESOURCES` |
| 확인한 캡처 | light, dark 파일명 |
| 통과한 검사 | typecheck, 금지색 grep, visual QA |
| 남은 위험 | API 미연결, 실제 Tauri 미검증 등 |

## 라우트 구현 우선순위

| 우선순위 | 라우트 | 이유 |
|---:|---|---|
| 1 | `/app` | Bubli가 무엇을 해주는지 가장 먼저 보이는 회원 앱 첫 화면 |
| 2 | `/app/project-rooms` | 사용자가 업무 단위를 고르는 출발점 |
| 3 | `/app/project-rooms/{roomId}` | 프로젝트룸 중심 구조를 보여주는 핵심 화면 |
| 4 | `/app/resources` | 자료가 업무로 바뀌는 흐름을 보여주는 화면 |
| 5 | `/app/project-rooms/{roomId}/resources` | 프로젝트룸 공유 자료 흐름 |
| 6 | `/app/project-rooms/{roomId}/work` | 후보가 확정 작업으로 바뀌는 화면 |
| 7 | `/app/project-rooms/{roomId}/chat` | 자료와 대화가 흩어지지 않는 흐름 |
| 8 | `/app/schedule` | 오늘 할 일과 일정 연결 |
| 9 | `/app/agent` | 에이전트 후보 검토 |
| 10 | `/app/desktop/widgets` | 데스크탑 버블 상태 검토 |

공개 사이트는 회원 앱 구조가 흔들리지 않는 선에서 따로 만든다.

## 화면 하나를 만들 때의 체크리스트

| 질문 | 통과 기준 |
|---|---|
| 이 화면은 어떤 사용자 문제를 푸는가 | 한 문장으로 설명된다 |
| 이 화면은 어떤 라우트인가 | 라우트 명세에 있다 |
| 이 화면 데이터의 원본은 어디인가 | API, Tauri IPC, 로컬 캐시가 구분된다 |
| 이 화면의 빈 상태는 무엇인가 | 사용자가 다음 행동을 안다 |
| 디자인보드에서 가져온 것은 무엇인가 | 색, 질감, 밀도만 가져왔다 |
| 빠진 v15 기능을 되살리지 않았는가 | `/signup`, 게스트, 링크 초대가 없다 |
| 캡처로 확인했는가 | light, dark 캡처가 있다 |

한마디로, 화면 구현은 "라우트 책임 확정 -> 데이터 흐름 확인 -> 낮은 충실도 배치 -> 상태 구현 -> 디자인 톤 적용 -> 캡처 검증" 순서로 간다.
