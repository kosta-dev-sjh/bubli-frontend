# 개인 대시보드 카드 후보 정리

작성일: 2026-07-01

## 기준

개인 대시보드는 공개 사이트나 소개 화면이 아니라 로그인한 사용자의 작업공간이다. 따라서 카드 후보는 오늘 바로 판단해야 하는 업무 상태, 여러 프로젝트룸에서 끌어온 개인 기준 요약, Tauri 버블에서 이어지는 개인 사용 흐름만 다룬다.

이번 참고 자료는 원본 화면을 복사하지 않고 표시 방식만 참고한다. 가져올 수 있는 구조는 다음 세 가지다.

- 원형 활동 지표: 오늘 할 일, 일정, 확인 필요, 실행 중 타이머, 버블 사용량, 집중 시간처럼 빠르게 숫자를 훑는 표현
- 반복 의도 목록: 자주 확인한 프로젝트룸, 자주 열린 자료, 반복해서 나온 확인 항목처럼 사용자가 계속 돌아오는 대상
- 누적 흐름: 작업 시작, 일정 처리, 확인 완료처럼 개인 작업 리듬이 이어지는 기록

제외하는 요소는 다음과 같다.

- 날씨, 오디오, 영양 정보, 운동 앱 성격의 항목
- 하단 마이크 버튼, 모바일 앱 원본 내비게이션, 장식성 모바일 구조
- 소개, 둘러보기, 시작하기 같은 랜딩식 문구
- 실제 API나 Tauri 런타임 근거 없이 항목이 있는 것처럼 보이는 데이터

## 카드 후보

| 후보 | 왜 필요한지 | 들어갈 데이터 | API 연결 후보 | 빈 상태 문구 | 우선순위 |
|---|---|---|---|---|---|
| 오늘 활동 링 | 대시보드 첫 화면에서 오늘 처리해야 할 양을 한눈에 본다. 기존 숫자 카드보다 밀도가 높고, 여러 프로젝트룸 데이터를 개인 기준으로 압축할 수 있다. | 할 일 수, 일정 수, 확인 필요 수, 실행 중 타이머 여부, 활성 버블 수 | `dashboardApi.getWork()`의 `todayTasks`, `todaySchedules`, `agentSuggestionSummary`, `runningTimer`; `widgetApi.getSummary()` | `오늘 표시할 항목이 없습니다` | 1 |
| 지금 이어 할 일 | 개인 대시보드의 첫 행동을 정한다. 프로젝트룸을 선택하지 않아도 오늘 가장 먼저 볼 작업을 보여준다. | 마감 임박 TODO, 진행 중 타이머의 대상, 오늘 일정 바로 전 항목 | `dashboardApi.getWork()`, `todoApi.list()` | `이어 할 일이 없습니다` | 1 |
| 자주 보는 프로젝트룸 | 사용자가 자주 드나드는 프로젝트룸으로 빠르게 돌아간다. 프로젝트룸 선택 흐름과 충돌하지 않고 개인 홈에 맞는 진입점이 된다. | 최근 열람 룸, 진행 중 룸, 오늘 할 일이 있는 룸 | `projectRoomApi.list()`, 향후 로컬 사용 기록 또는 위젯 사용 집계 | `자주 보는 프로젝트룸이 없습니다` | 2 |
| 자주 열린 자료 | 자료보드가 구글 드라이브처럼 보이되, 대시보드에서는 최근/반복 접근 자료만 짧게 보여준다. 개인 자료와 프로젝트룸 자료 범위를 함께 표시하되 자동 공유처럼 보이지 않게 구분한다. | 최근 개인 자료, 최근 프로젝트룸 자료, 다시 확인한 자료 | `resourcesApi.listPersonal()`, `resourcesApi.listRoomResources(roomId)`, 향후 최근 자료 전용 API | `최근에 연 자료가 없습니다` | 2 |
| 확인 필요 흐름 | 에이전트 후보와 사용자의 승인/보류 판단을 대시보드에서 놓치지 않게 한다. 작업판과 후보 화면으로 들어가기 전 개인 기준 큐 역할을 한다. | 확인할 후보 수, 반복된 확인 항목, 보류 중 후보 | `agentApi.listSuggestions()`, `agentApi.listRoomSuggestions(roomId)` | `확인할 항목이 없습니다` | 2 |
| 작업 연속 기록 | 단순 출석이 아니라 사용자가 실제 일을 이어간 흐름을 보여준다. 참고 자료의 streak/year texture는 작업 시작, 일정 처리, 확인 완료의 누적 히트맵으로만 변환한다. | 날짜별 작업 시작, 일정 완료, 후보 확인, 타이머 사용 | 향후 `activity` API, `widgetApi.getUsageSummariesToday()`, Tauri activity/usage rollup | `아직 누적 기록이 없습니다` | 3 |
| 버블 사용 요약 | Tauri 버블이 실제 작업에 쓰였는지 확인한다. 위젯은 웹앱 카드가 아니라 데스크탑 위젯이므로 대시보드에는 사용 요약만 둔다. | 열린 횟수, 상호작용 수, 표시 시간, 기기별 요약 | `widgetApi.getUsageSummariesToday()`, Tauri `rollup_widget_usage` | `오늘 버블 사용 기록이 없습니다` | 3 |
| 집중 시간 | 활동 인식이 켜진 경우에만 오늘 머문 작업 앱/창 흐름을 시간으로 요약한다. 사용자의 동의가 전제이며 화면 내용이나 키 입력은 다루지 않는다. | 활성 앱 이름, 창 제목, 머문 시간, 프로젝트룸 매핑 후보 | Tauri activity command, 향후 서버 activity API | `동의 후 표시됩니다` | 4 |

## 현재 연결 상태 반영

2026-07-01 기준 메인 브랜치 통합 후 대시보드에 반영된 연결은 다음과 같이 본다.

- `오늘 활동 링`: 대시보드 상단 요약 카드로 실제 반영됨. `dashboardApi.getWork()`와 `widgetApi.getSummary()`를 함께 사용한다.
- `지금 이어 할 일`: 대시보드 기본 카드로 실제 반영됨. 실행 중 타이머, 가장 먼저 볼 TODO, 다음 일정 순서로 묶어서 보여준다.
- 개인 TODO: `todoApi.list()`가 `PageResponse<TaskResponse>` 기준으로 정렬되었고, 대시보드에서 개인 할 일 후보로 연결 가능하다.
- 타이머: `DashboardWorkResponse.runningTimer` 타입이 백엔드 응답에 맞춰졌고, 대시보드에서 실행 중 타이머 표시가 가능하다.
- 위젯 summary/settings: `widgetApi` 응답 래핑이 백엔드 구조에 맞춰졌고, 대시보드 요약 카드 후보에서 사용할 수 있다.

아직 완료로 보지 않는 항목은 다음과 같다.

- `widget usage` 동기화 batch 계약 불일치
- Google Calendar 실제 OAuth 왕복 확인
- WebSocket 실서버 수신 확인
- Tauri 실제 앱 런타임 검증

## 구현 순서 제안

1. `오늘 활동 링`과 `지금 이어 할 일`을 기존 대시보드 카드의 표시 방식으로 먼저 반영한다.
2. `자주 보는 프로젝트룸`과 `자주 열린 자료`는 최근 사용 기록 API가 없으면 현재 리스트 API에서 보수적으로 표시하고, 최근성 정렬 근거를 문서에 남긴다.
3. `작업 연속 기록`, `버블 사용 요약`, `집중 시간`은 Tauri 런타임 검증과 사용량/활동 기록 계약을 먼저 잠근 뒤 카드로 올린다.
