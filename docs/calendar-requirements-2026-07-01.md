# 일정 화면 요구사항 정리 2026-07-01

## 기준

- 회원 웹 앱의 일정 화면은 소개 화면이 아니라 실제 일정을 확인하고 추가하는 작업 공간이다.
- 개인 일정, 현재 프로젝트룸 일정, 외부 캘린더 일정을 한 화면에서 볼 수 있어야 한다.
- 프로젝트룸을 선택한 상태에서는 그 룸의 일정과 룸 이벤트가 유지되어야 한다.
- 수동 새로고침은 기본 행동이 아니다. 데이터 변경은 API 재조회와 실시간 이벤트로 갱신하고, 실패 상태에서만 다시 연결을 제공한다.

## 화면에 있어야 하는 것

| 영역 | 필요한 기능 | 기준 |
|---|---|---|
| 월 보기 | 한 달 일정 밀도 확인 | 날짜별 일정 개수와 선택 상태가 보여야 한다 |
| 주 보기 | 이번 주 집중 확인 | 같은 데이터로 주 단위 스캔이 가능해야 한다 |
| 날짜 셀 | 실제 캘린더처럼 일정 제목 표시 | 숫자 카운트만 두지 않고 날짜 안에 일정 pill을 직접 보여준다 |
| 출처 필터 | 전체, 개인, 프로젝트룸, 외부 | 일정이 어디에서 온 것인지 사용자가 바로 구분한다 |
| 외부 캘린더 연결 | 연결 상태와 연결 버튼 | 헤더에 버튼만 덩그러니 두지 않고 상태 카드 안에 둔다 |
| 하루 타임라인 | 선택한 날짜 일정 목록 | 시간, 동기화 상태, 개인/프로젝트룸 구분을 보여준다 |
| 일정 추가 | 날짜 클릭 후 입력 창 | Mac 캘린더처럼 캘린더를 클릭하면 작은 입력 창이 뜨고, 캘린더 자체는 화면을 넓게 쓴다 |
| 룸 이벤트 | 일정과 작업 변경 기록 | 현재 프로젝트룸이 있을 때만 보조로 보여준다 |

## 제외할 것

- 기본 화면의 새로고침 버튼.
- 설명용 정책 카드 나열.
- Google Calendar 버튼만 헤더 우측에 크게 따로 두는 구조.
- 프로젝트룸 일정과 개인 일정을 구분하지 않는 목록.
- 캘린더 옆에 일정 추가 패널을 항상 붙여 캘린더 폭을 줄이는 구조.
- 캘린더 아래에 긴 보조 타임라인을 붙여 주 화면이 밀리는 구조.

## API 연결 기준

| 기능 | API | 현재 화면 연결 |
|---|---|---|
| 일정 목록 | `GET /api/schedules` | 월/주 보기, 하루 타임라인 |
| 일정 추가 | `POST /api/schedules` | 일정 추가 폼 |
| 일정 수정 | `PATCH /api/schedules/{scheduleId}` | 후속 편집 모드에서 사용 |
| 일정 삭제 | `DELETE /api/schedules/{scheduleId}` | 후속 편집 모드에서 사용 |
| 프로젝트룸 이벤트 | `GET /api/project-rooms/{roomId}/events` | 선택 날짜 룸 이벤트 |
| 외부 캘린더 연결 | `GET /api/calendar/google/connect` | 외부 캘린더 연결 버튼 |
